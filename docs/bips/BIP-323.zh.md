## 摘要

一种基于 JSON 的协议，用于描述 BTC 5 分钟预测市场上的自动化交易策略。一个策略是五个顺序决策的声明式描述：**过滤（filter）**、**信号（signal）**、**入场（entry）**、**风控（risk）**、**离场（exit）**。

## 动机

biubiu.tools 生态需要一种可移植、人类可读的策略定义格式，能在 UI 中创建、跨端点分享、在市场中发布、并由任何符合规范的引擎执行。现有的临时配置格式与特定实现紧密耦合，无法在系统间传输。

BIP-323 提供一个稳定的、有版本的 schema：

1. 覆盖 100% 的生产策略（27 个已验证）和 15 类真实交易策略。
2. 遵循交易员的心智模型：过滤 → 信号 → 入场 → 风控 → 离场。
3. 前向兼容：新增指标类型、过滤器类型和离场类型不会破坏现有策略。

## 规范

本文档中的关键词"必须（MUST）"、"不得（MUST NOT）"、"要求（REQUIRED）"、"应当（SHALL）"、"应该（SHOULD）"、"可以（MAY）"按照 RFC 2119 的定义解释。

### 1. 文档结构

一个符合规范的策略文档是一个 JSON 对象，包含以下顶层字段：

```json
{
  "$schema": "BIP-323",
  "id": "<string>",
  "name": "<string | {zh, en}>",
  "signal": { "method": "<string>", "params": {} },
  "entry": { "amount": "<number>", "method": "<string>" }
}
```

未列为"要求"的字段均为可选。实现方必须忽略未知字段。

### 2. 元数据字段

| 字段 | 类型 | 是否必须 | 约束 |
|------|------|----------|------|
| `$schema` | string | 必须 | 必须为 `"BIP-323"` |
| `id` | string | 必须 | 3-40 字符，仅允许 `[a-z0-9-]` |
| `name` | string \| {zh, en} | 必须 | 显示名称。字符串为单语言，对象为双语 |
| `description` | string \| {zh, en} | 可选 | 一句话摘要 |
| `tags` | string[] | 可选 | 分类标签 |
| `riskLevel` | string | 可选 | `"conservative"`、`"balanced"`、`"aggressive"` 之一 |
| `author` | string | 可选 | 创建者标识 |
| `version` | integer | 可选 | 策略迭代版本号，单调递增 |
| `createdAt` | string | 可选 | ISO 8601 日期时间 |

### 3. 过滤器（Filter）

可选。信号计算前评估的前置条件。所有已配置的过滤器必须全部通过，本轮才会继续。过滤器不决定方向。

```json
{
  "filter": {
    "schedule": { ... },
    "volatility": { ... },
    "trend": { ... }
  }
}
```

#### 3.1 schedule（时间计划）

| 字段 | 类型 | 是否必须 | 约束 |
|------|------|----------|------|
| `timezone` | string | 必须 | IANA 时区标识符 |
| `windows` | array | 必须 | 非空的时间窗口数组 |
| `windows[].days` | string[] | 必须 | 子集：`"mon"` `"tue"` `"wed"` `"thu"` `"fri"` `"sat"` `"sun"` |
| `windows[].from` | string | 必须 | `"HH:MM"` 格式，`from` < `to` |
| `windows[].to` | string | 必须 | `"HH:MM"` 格式 |

当前时间落入任意窗口内即通过。

#### 3.2 volatility（波动率）

| 字段 | 类型 | 是否必须 | 约束 |
|------|------|----------|------|
| `mode` | string | 必须 | `"require_high"` 或 `"require_low"` |
| `atrThreshold` | number | 必须 | 0 < atrThreshold < 1，ATR 占价格的比例 |

`"require_high"` 时，ATR% >= 阈值则通过。`"require_low"` 时，ATR% <= 阈值则通过。

#### 3.3 trend（趋势）

| 字段 | 类型 | 是否必须 | 约束 |
|------|------|----------|------|
| `minEfficiencyRatio` | number | 可选 | 0-1，Kaufman 效率比阈值 |
| `minBodyRatio` | number | 可选 | 0-1，平均 K 线实体/振幅比 |
| `lookback` | integer | 可选 | > 0，默认 10，回看 K 线数 |
| `mustMatchDirection` | boolean | 可选 | 默认 true，趋势方向必须与信号一致 |

### 4. 信号（Signal）

必须。决定交易方向（UP 或 DOWN）以及是否交易。

| 字段 | 类型 | 是否必须 |
|------|------|----------|
| `method` | string | 必须 |
| `params` | object | 取决于 method |

必须精确指定一种 method。

#### 4.1 clob_follow

跟随 CLOB 订单簿多数方向。无参数。

#### 4.2 clob_fade

CLOB 多数方向的反方向。无参数。

#### 4.3 fixed

固定方向交易。

| 参数 | 类型 | 是否必须 | 约束 |
|------|------|----------|------|
| `direction` | string | 必须 | `"up"` 或 `"down"` |

#### 4.4 candle

基于 K 线分析决定方向。三种模式。

| 参数 | 类型 | 是否必须 |
|------|------|----------|
| `mode` | string | 必须，`"follow"`、`"streak"`、`"template"` 之一 |

**mode: "follow"**

| 参数 | 类型 | 默认值 |
|------|------|--------|
| `reverse` | boolean | false。为 true 时，方向与上一根 K 线相反 |

**mode: "streak"**

| 参数 | 类型 | 是否必须 | 约束 |
|------|------|----------|------|
| `count` | integer | 必须 | 2-10，连续同向 K 线数 |
| `action` | string | 必须 | `"follow"`（顺势）或 `"reverse"`（反转） |

**mode: "template"**

| 参数 | 类型 | 是否必须 | 约束 |
|------|------|----------|------|
| `sequence` | CandleShape[] | 必须 | 1-128 个蜡烛形状 |
| `direction` | string | 必须 | `"up"`、`"down"` 或 `"reverse"` |
| `minSimilarity` | number | 可选 | 0-1，默认 0.75 |

CandleShape：

| 字段 | 类型 | 可选值 |
|------|------|--------|
| `direction` | string | `"bull"`、`"bear"`、`"any"` |
| `bodySize` | string | `"large"` (>70%)、`"medium"` (30-70%)、`"small"` (<30%)、`"any"` |
| `upperWick` | string | `"long"` (>50%)、`"short"`、`"any"` |
| `lowerWick` | string | `"long"`、`"short"`、`"any"` |

相似度计算方式：`匹配的非 any 特征数 / 总非 any 特征数`。设为 `"any"` 的特征不参与分子和分母的计算。若所有特征均为 `"any"`，则该蜡烛始终匹配。

当 `direction` 为 `"reverse"` 时：取匹配到的真实序列中最后一根蜡烛方向的反方向。

#### 4.5 rsi_reversal

| 参数 | 类型 | 是否必须 | 约束 |
|------|------|----------|------|
| `oversold` | number | 必须 | 0-100，RSI <= 此值 → UP |
| `overbought` | number | 必须 | 0-100，RSI >= 此值 → DOWN |

约束：`oversold` < `overbought`。RSI 在两个阈值之间时跳过本轮。

#### 4.6 scoring

多指标加权打分系统。

| 参数 | 类型 | 是否必须 |
|------|------|----------|
| `threshold` | number | 必须，总分 >= 此值则交易 |
| `allowReverse` | boolean | 可选，默认 false |
| `reverseThreshold` | number | 可选，总分 <= 此值则反向交易 |
| `indicators` | Indicator[] | 必须，1-20 个指标 |

Indicator：

| 字段 | 类型 | 是否必须 |
|------|------|----------|
| `name` | string | 必须 |
| `type` | string | 必须，见指标类型表 |
| `config` | object | 可选，类型特定参数 |
| `rules` | Rule[] | 必须，1 条以上评分规则 |
| `required` | boolean | 可选，为 true 时至少一条规则必须命中，否则跳过 |

Rule：

| 字段 | 类型 | 说明 |
|------|------|------|
| `when` | [number\|null, number\|null] | 值范围，null 表示无界，左闭右开 |
| `score` | number | 方向无关分数 |
| `up` | number | 评估 UP 方向时的分数 |
| `down` | number | 评估 DOWN 方向时的分数 |
| `veto` | boolean | 为 true 且范围命中时，立即跳过本轮 |

`score` 与 `up`/`down` 互斥。同时存在时 `up`/`down` 优先。

**指标类型：**

振荡器：

| 类型 | 配置 | 输出 |
|------|------|------|
| `rsi` | `{period: 14}` | 0-100 |
| `stochastic` | `{kPeriod: 14, dPeriod: 3}` | 0-100 |
| `macd` | `{fast: 12, slow: 26, signal: 9}` | 实数 |

趋势：

| `ema_crossover` | `{fast: 9, slow: 21}` | 0 或 1 |
| `price_vs_ema` | `{period: 21}` | 百分比 |
| `adx` | `{period: 14}` | 0-100 |
| `efficiency_ratio` | `{period: 15}` | 0-1 |

波动率：

|------|------|------|

量能：

#### 4.7 ml

| 参数 | 类型 | 是否必须 |
|------|------|----------|

| `type` | string | 必须，`"random_forest"`、`"xgboost"` 或 `"gbdt"` |
| `minConfidence` | number | 可选，0.5-1.0，默认 0.6 |

### 5. 入场（Entry）

必须。执行参数。

| 字段 | 类型 | 是否必须 | 约束 |
|------|------|----------|------|
| `amount` | number | 必须 | 0.01-10000，每轮 USD |
| `method` | string | 必须 | `"market"`、`"limit"`、`"swing"` 或 `"scan"` |
| `maxPrice` | number | 可选 | 0.01-0.99，最高 share price |
| `minPrice` | number | 可选 | 0.01-0.99，默认 0.01，最低 share price |
| `windows` | Window[] | 可选 | 1-3 个入场时间窗口 |
| `maxPerHour` | integer | 可选 | 频率限制 |
| `sizing` | Sizing | 可选 | 动态仓位管理 |

Window：

| 字段 | 类型 | 约束 |
|------|------|------|
| `from` | integer | 窗口开启时的剩余秒数，`from` > `to` |
| `to` | integer | 窗口关闭时的剩余秒数，>= 0 |

各入场方式的附加字段：

| 方式 | 附加字段 |
|------|----------|
| `market` | 无 |
| `limit` | `limitPrice`（必须）、`timeoutSec`（可选） |
| `swing` | `targetPrice`（必须）、`windowSec`（可选） |
| `scan` | `scanMaxPrice`、`scanBudget`、`scanDelaySec`、`scanDurationSec`、`scanIntervalMs`（均必须） |

Sizing：

| 字段 | 类型 | 约束 |
|------|------|------|
| `method` | string | `"fixed"`、`"martingale"`、`"anti_martingale"` 或 `"kelly"` |
| `multiplier` | number | martingale/anti_martingale 必须 |
| `maxAmount` | number | 调整后的上限 |
| `minAmount` | number | 最低金额（kelly 用） |
| `resetOnWin` | boolean | martingale：赢了回到基础金额 |
| `resetOnLoss` | boolean | anti_martingale：亏了回到基础金额 |
| `lookback` | integer | kelly：用于计算胜率的轮数 |
| `fraction` | number | kelly：最优值的折扣（0.5 = 半 Kelly） |

未配置 `sizing` 时，金额固定不变。

### 6. 风控（Risk）

可选。保护机制。

| 字段 | 类型 | 说明 |
|------|------|------|
| `dailyLossLimit` | number | 每日最大 USD 亏损，触达后暂停 |
| `streakRules` | StreakRule[] | 连胜/连败行为规则 |
| `hedge` | Hedge | 反方向对冲保险 |
| `stopLoss` | StopLoss | 价格暴跌时提前离场 |

StreakRule：

| 字段 | 类型 | 是否必须 | 约束 |
|------|------|----------|------|
| `trigger` | string | 必须 | `"win"` 或 `"loss"` |
| `count` | integer | 必须 | >= 2 |
| `action` | string | 必须 | `"pause"`、`"stop"` 或 `"adjust_amount"` |
| `pauseMinutes` | integer | action=pause 时 | 1-1440 |
| `multiply` | number | action=adjust_amount 时 | 金额乘数 |
| `maxAmount` | number | action=adjust_amount 时 | 上限 |

Hedge：

| 字段 | 类型 | 是否必须 |
|------|------|----------|
| `enabled` | boolean | 必须 |
| `price` | number | 必须，0.001-0.99 |
| `shares` | integer | 必须，1-10000 |
| `sellAbove` | number | 必须，卖出阈值 |

StopLoss：

| 字段 | 类型 | 是否必须 |
|------|------|----------|
| `price` | number | 必须，触发价格 |
| `minHoldSec` | integer | 可选，止损生效前的最短持有时间 |

### 7. 离场（Exit）

可选。按顺序评估的离场规则数组，首个触发的生效。缺省或为空时，隐含 `[{"type": "settlement"}]`。

| 类型 | 字段 | 触发条件 |
|------|------|----------|
| `takeProfit` | `profitPct`（必须，0-1） | 持仓利润 >= profitPct |
| `trailingStop` | `drawdownPct`（必须，0-1） | 从最高点回撤 >= drawdownPct |
| `checkpoint` | `atRemainingSec`（必须，> 0） | 在剩余 N 秒时重新评估信号，方向反转则离场 |
| `settlement` | 无 | 等待本轮结算 |

实现方必须在数组末尾追加 `settlement`（如果不存在）。

### 8. 验证规则

符合规范的实现方必须拒绝违反以下任何规则的策略文档：

1. `$schema` 必须为 `"BIP-323"`。
2. `id` 必须匹配 `^[a-z0-9-]{3,40}$`。
3. `signal.method` 必须是已识别的方法，否则必须拒绝文档。
4. `signal.method = "post_close"` 要求 `entry.method = "scan"` 且 `entry.windows = []`。
5. `signal.method = "rsi_reversal"` 要求 `params.oversold < params.overbought`。
6. `signal.method = "ml"` 要求 `params.model` 和 `params.type` 均存在。
7. `signal.method = "candle"` 且 `mode = "template"` 要求 `params.sequence` 含 1-128 项。
8. `signal.method = "fixed"` 要求 `params.direction` 存在。
9. `entry.method = "limit"` 要求 `entry.limitPrice` 存在。
10. `entry.method = "swing"` 要求 `entry.targetPrice` 存在。
11. `entry.method = "scan"` 要求 `scanMaxPrice`、`scanBudget`、`scanDelaySec`、`scanDurationSec`、`scanIntervalMs` 均存在。
12. `entry.windows[].from` 必须 > `entry.windows[].to`。
13. 当两者同时存在时，`entry.minPrice` 必须 < `entry.maxPrice`。
14. `risk.stopLoss.price` 应该 < `entry.maxPrice`。
15. `filter.schedule.windows[].from` 必须 < `windows[].to`（按时间顺序）。
16. 实现方必须忽略未知字段。

### 9. 最小有效文档

```json
{
  "$schema": "BIP-323",
  "id": "simple-follow",
  "name": "Simple Follow",
  "signal": { "method": "clob_follow" },
  "entry": { "amount": 5, "method": "market" }
}
```

### 10. 扩展性

未来 BIP 可以在不修改本规范的前提下新增指标类型、过滤器类型和离场类型。实现方在 `scoring` 信号中遇到未知指标类型时，应该跳过该指标继续执行。实现方遇到未知的 `signal.method` 或 `entry.method` 时，必须拒绝文档。

未来 BIP 可以新增顶层字段。实现方必须按规则 16 忽略未知顶层字段。

## 设计理由

**为什么是五个部分？** filter→signal→entry→risk→exit 管线对应交易员每轮做出的顺序决策。其他结构（如将对冲归入入场，或将离场归入风控）曾被考虑但被否决，因为它们混淆了不同关注点。

**为什么用 scoring 而不是多种信号方法？** 每个策略只允许一种信号方法，避免了组合冲突信号时的歧义。`scoring` 方法通过加权指标提供了有原则的组合机制。

**为什么用 streakRules 而不是 cooldown？** 简单的冷却只处理连败。真实交易员也管理连胜（减少过度自信）并根据近期表现调整仓位。`streakRules` 用触发/动作模式将其泛化。

**为什么用 candle.template 而不是命名模式？** 命名模式（如"锤子线"、"吞没"）是主观的——不同交易员定义不同。template 方法将模式分解为客观特征（方向、实体大小、影线长度），可以精确指定和匹配。

**为什么 minSimilarity 排除 "any" 特征？** 如果 `"any"` 特征计入分母，大量通配符的模板会产生虚高的相似度分数，使阈值失去意义。
