## Abstract

Defines the runtime input/output contract for executing strategies described by BIP-323. Specifies the data an engine MUST provide each round (RoundContext), the decision format a strategy evaluator MUST return (SignalOutput), and the standardized interface for ML model inference (ModelInterface).

## Motivation

BIP-323 defines what a strategy *is*. This BIP defines how a strategy *runs*. Without a standardized runtime interface, two conforming engines could provide different data to the same strategy and produce different results. BIP-324 ensures deterministic execution: given the same strategy and the same market data, any conforming engine MUST produce the same trading decision.

## Specification

### 1. Execution Lifecycle

A single round consists of 6 phases. The engine MUST execute them in order:

```
Phase 1: Round Discovery     â†’ engine detects new round, creates RoundContext
Phase 2: Data Collection      â†’ engine polls market data, updates RoundContext
Phase 3: Strategy Evaluation  â†’ engine feeds RoundContext to evaluator, receives SignalOutput
Phase 4: Order Execution      â†’ engine executes trade based on SignalOutput (out of scope)
Phase 5: Position Monitoring  â†’ engine monitors exits, produces ExitSignal if triggered
Phase 6: Settlement           â†’ round ends, engine records outcome
```

Phases 2-3 repeat on each data update (typically every 1-3 seconds) until entry or round end. Phase 5 repeats on each data update after entry until exit or settlement.

### 2. RoundContext

The data object an engine MUST construct and pass to the strategy evaluator.

```typescript
interface RoundContext {
  // â”€â”€ Round Identity â”€â”€
  roundId: string;                     // Unique round identifier
  remainingSec: number;                // Seconds until round settlement

  // â”€â”€ CLOB Data (REQUIRED for all methods) â”€â”€
  clob: {
    upBestBid: number;
    upBestAsk: number;
    upMid: number;                     // (bestBid + bestAsk) / 2
    downBestBid: number;
    downBestAsk: number;
    downMid: number;
```

#### 2.4 MacroEventContext

A macro event is not a single timestamp â€” it has multiple **phases** with different market impacts. The engine constructs this from a cached calendar, and the strategy creator or analyst defines the phases.

```typescript
interface MacroEventContext {
  id: string;                          // Unique event identifier
  name: string;                        // e.g., "US CPI (Feb 2026)"
  category: string;                    // "fomc" | "cpi" | "nfp" | "jobless" | "gdp" | "custom"

  // â”€â”€ Current phase (engine computes from phases[] + current time) â”€â”€
  currentPhase: string | null;         // null = outside all phases
  minutesInPhase: number;              // How long into current phase (negative = before phase start)

  // â”€â”€ Phase definitions (from calendar, editable by analyst) â”€â”€
  phases: MacroEventPhase[];

  // â”€â”€ Event metadata â”€â”€
  previousValue?: string;              // e.g., "3.1%" (last CPI reading)
  expectedValue?: string;              // e.g., "2.9%" (consensus forecast)
  actualValue?: string;                // e.g., "3.2%" (filled after release)
}

interface MacroEventPhase {
  name: string;                        // Phase identifier
  start: string;                       // ISO 8601 datetime
  end: string;                         // ISO 8601 datetime
  impact: "low" | "medium" | "high" | "extreme";
  description?: string;                // Analyst notes
}
```

**Standard phases for a major event (e.g., CPI):**

| Phase | Typical Timing | Impact | Market Behavior |
|-------|---------------|--------|-----------------|
| `"buildup"` | 3-7 days before | medium | Positioning, IV rising, reduced liquidity |
| `"anticipation"` | 1-24 hours before | high | Spread widening, cautious trading |
| `"release_window"` | 5 min before â†’ 5 min after | extreme | Maximum volatility spike |
| `"digestion"` | 5-60 min after | high | Market finds direction, trend emerges |
| `"aftermath"` | 1 hour â†’ 2 days after | medium | Secondary effects, related events reprice |

Example calendar entry (analyst-edited):

```json

  "name": "US CPI (Feb 2026)",
  "category": "cpi",
  "previousValue": "3.1%",
  "expectedValue": "2.9%",
  "phases": [
    { "name": "buildup",        "start": "2026-03-09T00:00:00Z", "end": "2026-03-11T00:00:00Z", "impact": "medium", "description": "Analysts publishing forecasts" },
    { "name": "anticipation",   "start": "2026-03-11T00:00:00Z", "end": "2026-03-12T12:25:00Z", "impact": "high" },
    { "name": "release_window", "start": "2026-03-12T12:25:00Z", "end": "2026-03-12T12:35:00Z", "impact": "extreme", "description": "CPI released at 12:30 UTC" },
    { "name": "digestion",      "start": "2026-03-12T12:35:00Z", "end": "2026-03-12T13:30:00Z", "impact": "high", "description": "Market digests data, direction clarifies" },
    { "name": "aftermath",      "start": "2026-03-12T13:30:00Z", "end": "2026-03-14T00:00:00Z", "impact": "medium", "description": "FOMC rate expectations reprice based on CPI" }
  ]

```

**Engine runtime behavior:**

1. Engine fetches calendar from `biubiu.tools/api/macro-calendar` on startup, caches locally, refreshes daily.
2. Each round, engine scans cached events for any phase containing current time.
3. If found: populates `roundContext.macroEvent` with event data + computed `currentPhase` and `minutesInPhase`.
4. If none: sets `macroEvent = null`.

**Role in the system:**

macroEvent is NOT referenced by BIP-323 strategy filters. Macro events are one-time occurrences â€” hardcoding them into reusable strategy rules would make strategies brittle and non-portable. Strategies that need to handle high-volatility periods SHOULD use `filter.volatility` instead, which is event-agnostic and always works.

macroEvent in RoundContext serves three purposes:

1. **Analytics** â€” Post-hoc analysis: "your strategy lost $80 during FOMC. Consider adding a volatility filter."
2. **Backtesting** â€” Tag historical rounds with macro context to evaluate strategy behavior around events.
3. **UI guidance** â€” Strategy editor can surface warnings: "CPI is in 2 hours. Your strategy has no volatility filter."

**Calendar maintenance:**

biubiu.tools provides the API and a calendar editor UI. Analysts can:
- Add events from known schedules (FOMC, CPI, NFP dates are published annually)
- Define custom phases per event
- Edit `expectedValue` as consensus updates
- Fill `actualValue` after release

Individual endpoints cache the calendar locally. The calendar is reference data, not real-time data â€” a daily refresh is sufficient.

### 3.1 Signal Method Evaluation Rules

For each signal method defined in BIP-323 Section 4, the engine MUST evaluate direction as follows:

#### clob_follow / clob_fade

```
let upAsk = clob.upBestAsk
let downAsk = clob.downBestAsk

if upAsk or downAsk is unavailable â†’ skip "no_liquidity"
if upAsk < downAsk  â†’ majorityDirection = "up"
if downAsk < upAsk  â†’ majorityDirection = "down"
if upAsk == downAsk â†’ skip "no_majority"

clob_follow â†’ direction = majorityDirection
clob_fade   â†’ direction = opposite(majorityDirection)
```

The side with the lower ask price (cheaper to buy) is the "majority direction" â€” it reflects which outcome the market collectively favors. Ask price is used instead of midpoint because it represents the actionable entry price.

Direction and price filtering (`entry.minPrice`, `entry.maxPrice`) are independent concerns evaluated in separate steps (Section 7, steps 8-9).

#### candle (follow)

```
let last = klines[klines.length - 1]
if last.close > last.open â†’ candleDir = "up"
if last.close < last.open â†’ candleDir = "down"
if last.close == last.open â†’ skip "no_signal"

if params.reverse â†’ direction = opposite(candleDir)
else              â†’ direction = candleDir
```

#### candle (streak)

```
let recent = last params.count candles
let dirs = recent.map(k â†’ k.close > k.open ? "up" : "down")
if not all dirs are the same â†’ skip "no_signal"

if params.action == "follow"  â†’ direction = dirs[0]
if params.action == "reverse" â†’ direction = opposite(dirs[0])
```

#### candle (template)

```
let n = params.sequence.length
let recent = last n candles
let totalNonAny = 0, matched = 0

for each candle i in 0..n:
  let tpl = params.sequence[i]
  let actual = classify(recent[i])
  for feature in [direction, bodySize, upperWick, lowerWick]:
    if tpl[feature] == "any" â†’ continue
    totalNonAny++
    if tpl[feature] == actual[feature] â†’ matched++

similarity = totalNonAny == 0 ? 1.0 : matched / totalNonAny
if similarity < params.minSimilarity â†’ skip "template_no_match"

if params.direction == "up" or "down" â†’ direction = params.direction
if params.direction == "reverse" â†’ direction = opposite(classify(recent[n-1]).direction)
```

**classify(kline):**

| Feature | Classification |
|---------|---------------|
| direction | close > open â†’ "bull", close < open â†’ "bear", else â†’ "any" |
| bodySize | body/range > 0.7 â†’ "large", > 0.3 â†’ "medium", else â†’ "small" |
| upperWick | (high - max(open,close)) / range > 0.5 â†’ "long", else â†’ "short" |
| lowerWick | (min(open,close) - low) / range > 0.5 â†’ "long", else â†’ "short" |

When range (high - low) is 0, bodySize = "small", wicks = "short".

#### rsi_reversal

```
if indicators.rsi14 is null â†’ skip "data_unavailable"
if rsi14 <= params.oversold   â†’ direction = "up"
if rsi14 >= params.overbought â†’ direction = "down"
else â†’ skip "no_signal"
```

#### scoring

```
for each indicator:
  value = compute(indicator.type, indicator.config, klines)
  if value is null:
    if indicator.required â†’ skip "signal_veto"
    else â†’ continue

  for each rule (first match wins):
    if value in rule.when range:
      if rule.veto â†’ skip "signal_veto"
      scoreUp   += rule.up   ?? rule.score ?? 0
      scoreDown += rule.down ?? rule.score ?? 0
      break

if scoreUp >= threshold AND scoreUp >= scoreDown â†’ direction = "up"
elif scoreDown >= threshold AND scoreDown > scoreUp â†’ direction = "down"
elif allowReverse AND scoreUp <= reverseThreshold â†’ direction = "down"
elif allowReverse AND scoreDown <= reverseThreshold â†’ direction = "up"
else â†’ skip "no_signal"
```

#### fixed

```
direction = params.direction   // Always enters, never skips from signal
```

#### ml

```
features = computeFeatures(roundContext)   // Per Section 6.2
prediction = model.predict(features)       // â†’ ModelOutput
if prediction.confidence < params.minConfidence â†’ skip "low_confidence"
direction = prediction.direction
```

#### post_close

```
// After market close, poll CLOB midpoints
loop until params.timeoutSec elapsed:
  if clob.upMid >= params.confidencePrice â†’ direction = "up", break
  if clob.downMid >= params.confidencePrice â†’ direction = "down", break
  wait(pollInterval)
else â†’ skip "no_signal"
```

### 4. SignalOutput

The strategy evaluator MUST return a SignalOutput for each evaluation cycle.

```typescript
interface SignalOutput {
  action: "enter" | "skip";
  direction?: "up" | "down";          // REQUIRED when action = "enter"
  confidence?: number;                 // 0-1, OPTIONAL
  skipReason?: string;                 // REQUIRED when action = "skip"

  // â”€â”€ Diagnostics (OPTIONAL, for logging/UI) â”€â”€
  diagnostics?: {
    signalMethod: string;              // Method that produced this decision
    score?: number;                    // Scoring total (for scoring method)
    indicators?: {                     // Individual indicator results
      name: string;
      value: number | null;
      score: number;
    }[];
    modelProbUp?: number;              // ML model probabilities
    modelProbDown?: number;

      passed: boolean;

```

#### 4.1 Skip Reasons

| Reason | Meaning |
|--------|---------|
| `"no_signal"` | Signal method produced no actionable signal (e.g., RSI in neutral zone) |
| `"price_above_max"` | Best ask > entry.maxPrice |
| `"price_below_min"` | Best ask < entry.minPrice |
| `"outside_window"` | Not within any entry time window |
| `"max_per_hour"` | Hourly round limit reached |
| `"daily_loss_limit"` | Daily loss limit reached |
| `"streak_pause"` | Paused due to streak rule |
| `"filter_schedule"` | Outside scheduled trading hours |
| `"filter_volatility"` | Volatility condition not met |
| `"filter_trend"` | Trend condition not met |
| `"signal_veto"` | Indicator veto triggered |
| `"low_confidence"` | ML model confidence below threshold |
| `"template_no_match"` | Candle template similarity below threshold |

### 5. ExitSignal

When a position is held, the engine evaluates exit rules on each data update. If triggered:

```typescript
interface ExitSignal {
  action: "exit" | "hold";
  exitType?: string;                   // REQUIRED when action = "exit"
  reason?: string;

  // For position monitoring
  currentPrice?: number;
  currentProfit?: number;
  drawdownFromPeak?: number;
}
```

| exitType | Trigger |
|----------|---------|
| `"takeProfit"` | Profit % >= profitPct |
| `"trailingStop"` | Drawdown from highWaterMark >= drawdownPct |
| `"checkpoint"` | Signal re-evaluation reversed direction |
| `"stopLoss"` | Price <= risk.stopLoss.price AND held >= minHoldSec |
| `"settlement"` | Round ended, position auto-settled |

#### 6.3 ModelOutput

```typescript
interface ModelOutput {
  direction: "up" | "down";
  confidence: number;                  // 0.5-1.0
  probUp: number;                      // Probability of UP
  probDown: number;                    // Probability of DOWN
}
```

`confidence` = max(probUp, probDown). `direction` = probUp > probDown ? "up" : "down".

The engine MUST skip the round if `confidence < params.minConfidence`.

#### 6.4 Model Portability

A model file MUST be self-contained. It MUST NOT depend on external state beyond the feature vector defined in 6.2. The model file format is implementation-specific (JSON, ONNX, pickle, etc.) but the I/O contract defined in 6.1-6.3 is universal.

When a model is published to the marketplace (BIP-323 `params.model` with `market:` prefix), the feature categories it requires MUST be documented alongside the model file. An engine that cannot compute a required feature category MUST reject the strategy with error `"unsupported_features"`.

### 7. Evaluation Order

When evaluating a BIP-323 strategy against a RoundContext, the engine MUST follow this order:

```
1. Check state.roundsThisHour < entry.maxPerHour     â†’ skip "max_per_hour"
2. Check state.dailyPnl > -risk.dailyLossLimit       â†’ skip "daily_loss_limit"
3. Check risk.streakRules against state               â†’ skip "streak_pause"
4. Check filter.schedule against current time         â†’ skip "filter_schedule"
5. Check filter.volatility against indicators.atrPct  â†’ skip "filter_volatility"
6. Check filter.trend against klines                  â†’ skip "filter_trend"
7. Check entry.windows against remainingSec           â†’ skip "outside_window"
8. Evaluate signal.method â†’ direction + confidence
9. Check entry.minPrice <= bestAsk <= entry.maxPrice  â†’ skip "price_above/below"
10. Return SignalOutput { action: "enter", direction }
```

Steps 1-7 are pre-checks. If any fails, return `skip` immediately without evaluating the signal. This ensures expensive computations (signal scoring, ML inference) are only performed when all pre-conditions pass.

### 8. Error Handling

| Error | Behavior |
|-------|----------|
| `klines` unavailable when required | Skip round with reason `"data_unavailable"` |
| `trades` unavailable when required | Skip round with reason `"data_unavailable"` |
| Indicator computation returns null | Skip the indicator in scoring; skip round for rsi_reversal |
| ML model inference fails | Skip round with reason `"model_error"` |
| Unknown indicator type in scoring | Skip that indicator, continue with remaining |

Errors MUST NOT crash the engine. All errors MUST be logged and reported via the diagnostics field.

## Rationale

**Why RoundContext instead of raw market data?** Pre-structuring the data into a typed context ensures all engines provide the same information. Without this, engine A might provide 30 candles while engine B provides 50, leading to different RSI values.

**Why standardized feature names for ML?** Models trained on one engine must work on another. If feature names or computations differ, the model produces garbage. The `k_`/`t_`/`p_`/`h_`/`time_` prefix convention makes features self-documenting.

**Why fixed evaluation order?** Cheap checks (state, schedule) before expensive ones (signal scoring, ML inference) is an optimization that all engines should follow. Making it mandatory ensures consistent skip reasons.

**Why diagnostics in SignalOutput?** Without diagnostics, debugging a strategy is impossible. "Why did it skip?" needs a structured answer, not just "it skipped." The diagnostics field is OPTIONAL for production but invaluable for development.

## Security Considerations

Engines MUST NOT pass RoundContext data to untrusted external services without user consent. The `trades` field contains market microstructure data that could be commercially sensitive. ML model inference MUST be performed locally; the model file MUST NOT phone home.
