## 摘要

定义执行 BIP-323 策略时的运行时输入/输出契约。规定引擎每轮必须提供的数据（RoundContext）、策略评估器必须返回的决策格式（SignalOutput）、以及 ML 模型推理的标准化接口（ModelInterface）。

## 动机

BIP-323 定义策略*是什么*。本 BIP 定义策略*怎么跑*。若无标准化的运行时接口，两个符合规范的引擎可能向同一策略提供不同数据，产生不同结果。BIP-324 确保确定性执行：给定相同策略和相同市场数据，任何符合规范的引擎必须产出相同的交易决策。

## 规范

### 1. 执行生命周期

单轮包含 6 个阶段，引擎必须按顺序执行：

```
阶段 1: 轮次发现     → 引擎检测到新轮次，创建 RoundContext
阶段 2: 数据收集     → 引擎轮询市场数据，更新 RoundContext
阶段 3: 策略评估     → 引擎将 RoundContext 传入评估器，接收 SignalOutput
阶段 4: 订单执行     → 引擎根据 SignalOutput 执行交易（不在本 BIP 范围内）
阶段 5: 持仓监控     → 引擎监控离场条件，触发时产出 ExitSignal
阶段 6: 结算         → 轮次结束，引擎记录结果
```

阶段 2-3 在每次数据更新时重复（通常每 1-3 秒），直到入场或轮次结束。阶段 5 在入场后每次数据更新时重复，直到离场或结算。

### 2. RoundContext

引擎必须构造并传递给策略评估器的数据对象。

```typescript
interface RoundContext {
  // ── 轮次身份 ──
  roundId: string;                     // 唯一轮次标识
  remainingSec: number;                // 距结算的剩余秒数

  // ── CLOB 数据（所有方法均需要）──
  clob: {
    upBestBid: number;
    upBestAsk: number;
    upMid: number;                     // (bestBid + bestAsk) / 2
    downBestBid: number;
    downBestAsk: number;
    downMid: number;

    // ── CLOB 流数据（可选，从订单簿 + 成交流派生）──
    obi?: number;                      // 订单簿不平衡度 [-1,1]: (bidVol-askVol)/(bidVol+askVol)
    tradeFlow?: number;                // 成交方向加权 [0,1]: 买入成交量 / 总成交量
    largeTradeCount?: number;          // 大单数量（>$500）
    largeTradeDir?: number;            // 大单方向 [-1,1]: 大单净方向
    priceDivergence?: number;          // Polymarket 中间价与 Binance 隐含公允值的偏差（美分）
  };

  // ── Binance 现货数据 ──
  // 当策略使用 candle、rsi_reversal、scoring 或 ml 时必须提供
  // klines 必须包含至少 30 根已完成的 1 分钟 K 线
  klines?: Kline[];

  // ── Binance 实时数据（可选，来自 WebSocket）──
  binance?: {
    vwap5m: number | null;             // 5 分钟成交量加权均价
    bidAskImbalance: number | null;    // 20 档深度不平衡 [-1,1]
    microMomentum: number | null;      // 5 秒价格加速度（二阶导数）
    lastPrice: number | null;          // 最新成交价
  };

  // ── 预计算指标 ──
  // 当提供 klines 时必须同时提供
  indicators?: {
    rsi14: number | null;              // 基于收盘价的 RSI(14)
    atr14: number | null;              // ATR(14)
    atrPct: number | null;             // atr14 / 最新收盘价
  };

  // ── 交易数据 ──
  // 当策略使用 ml 时必须提供
  trades?: Trade[];

  // ── 期货数据（可选，来自 Binance Futures API）──
  futures?: {
    fundingRate: number | null;        // 当前资金费率（如 0.0001 = 0.01%）
    oiChange: number | null;           // 5 分钟内未平仓量变化百分比
    longShortRatio: number | null;     // 散户多空账户比
    takerRatio: number | null;         // 期货主动买卖比 [0,1]
  };

  // ── 关联资产（可选，来自 Binance 公开 WebSocket）──
  correlatedAssets?: {
    ethBtcChange: number | null;       // ETH/BTC 5 分钟变化百分比（Binance ETHBTC stream）
  };

  // ── 宏观事件（可选）──
  // 引擎从 biubiu.tools/api/macro-calendar 获取日历，缓存到本地。
  // 事件有多个阶段，每个阶段有不同的市场影响。
  macroEvent?: MacroEventContext | null;

  // ── 历史结果 ──
  // 当策略使用 ml 或 entry.sizing (kelly) 时必须提供
  history?: {
    outcomes: ("up" | "down")[];       // 最近 20 轮结果，最新在前
    results: ("win" | "loss")[];       // 本策略最近 20 轮胜负
  };

  // ── 策略状态 ──
  state: {
    dailyPnl: number;                  // 今日累计盈亏（USD）
    consecutiveWins: number;           // 当前连胜数（上次为亏则为 0）
    consecutiveLosses: number;         // 当前连败数（上次为赢则为 0）
    roundsThisHour: number;            // 本小时已入场轮数
    currentPosition: Position | null;  // 持仓中时非 null
  };
}
```

#### 2.4 MacroEventContext（宏观事件上下文）

宏观事件不是一个时间点，而是有多个**阶段**的时间线，每个阶段对市场影响不同。引擎从缓存的日历中构造此对象，日历由策略创建者或分析师定义各阶段。

```typescript
interface MacroEventContext {
  id: string;                          // 唯一事件标识
  name: string;                        // 如 "US CPI (Feb 2026)"
  category: string;                    // "fomc" | "cpi" | "nfp" | "jobless" | "gdp" | "custom"

  // ── 当前阶段（引擎根据 phases[] + 当前时间计算）──
  currentPhase: string | null;         // null = 不在任何阶段内
  minutesInPhase: number;              // 进入当前阶段多久（负数 = 尚未开始）

  // ── 阶段定义（来自日历，分析师可编辑）──
  phases: MacroEventPhase[];

  // ── 事件元数据 ──
  previousValue?: string;              // 如 "3.1%"（上次 CPI 数据）
  expectedValue?: string;              // 如 "2.9%"（市场共识预期）
  actualValue?: string;                // 如 "3.2%"（发布后填入）
}

interface MacroEventPhase {
  name: string;                        // 阶段标识
  start: string;                       // ISO 8601 开始时间
  end: string;                         // ISO 8601 结束时间
  impact: "low" | "medium" | "high" | "extreme";
  description?: string;                // 分析师备注
}
```

**重大事件的标准阶段（如 CPI）：**

| 阶段 | 典型时机 | 影响 | 市场表现 |
|------|----------|------|----------|
| `"buildup"` | 事件前 3-7 天 | medium | 机构建仓，IV 上升，流动性下降 |
| `"anticipation"` | 事件前 1-24 小时 | high | 价差扩大，交易谨慎 |
| `"release_window"` | 事件前 5 分钟 → 后 5 分钟 | extreme | 最大波动率高峰 |
| `"digestion"` | 事件后 5-60 分钟 | high | 市场消化数据，趋势显现 |
| `"aftermath"` | 事件后 1 小时 → 2 天 | medium | 二次效应，关联事件重新定价 |

日历条目示例（分析师编辑）：

```json
{
  "id": "cpi-2026-03",
  "name": "US CPI (Feb 2026)",
  "category": "cpi",
  "previousValue": "3.1%",
  "expectedValue": "2.9%",
  "phases": [
    { "name": "buildup",        "start": "2026-03-09T00:00:00Z", "end": "2026-03-11T00:00:00Z", "impact": "medium", "description": "分析师发布预测" },
    { "name": "anticipation",   "start": "2026-03-11T00:00:00Z", "end": "2026-03-12T12:25:00Z", "impact": "high" },
    { "name": "release_window", "start": "2026-03-12T12:25:00Z", "end": "2026-03-12T12:35:00Z", "impact": "extreme", "description": "CPI 于 12:30 UTC 发布" },
    { "name": "digestion",      "start": "2026-03-12T12:35:00Z", "end": "2026-03-12T13:30:00Z", "impact": "high", "description": "市场消化数据，方向明确" },
    { "name": "aftermath",      "start": "2026-03-12T13:30:00Z", "end": "2026-03-14T00:00:00Z", "impact": "medium", "description": "FOMC 加息预期基于 CPI 重新定价" }
  ]
}
```

**引擎运行时行为：**

1. 引擎启动时从 `biubiu.tools/api/macro-calendar` 获取日历，缓存到本地，每日刷新。
2. 每轮扫描缓存的事件，查找包含当前时间的阶段。
3. 如找到：填充 `roundContext.macroEvent`，包括事件数据 + 计算出的 `currentPhase` 和 `minutesInPhase`。
4. 如无：设 `macroEvent = null`。

**在系统中的角色：**

macroEvent 不被 BIP-323 策略过滤器引用。宏观事件是一次性的——将其硬编码到可复用的策略规则中会使策略脆弱且不可移植。需要处理高波动时段的策略应该使用 `filter.volatility`，它与事件无关，始终有效。

RoundContext 中的 macroEvent 有三个用途：

1. **分析** — 事后复盘："你的策略在 FOMC 期间亏了 $80，建议加一个波动率过滤器。"
2. **回测** — 用宏观事件标记历史轮次，评估策略在事件前后的表现。
3. **UI 引导** — 策略编辑器提示："CPI 将在 2 小时后发布，你的策略没有波动率过滤器。"

**日历维护：**

biubiu.tools 提供 API 和日历编辑器 UI。分析师可以：
- 从已知日程添加事件（FOMC、CPI、NFP 日期每年提前公布）
- 为每个事件定义自定义阶段
- 随共识更新编辑 `expectedValue`
- 发布后填入 `actualValue`

各端点将日历缓存到本地。日历是参考数据，不是实时数据——每日刷新即可。

### 3. 各信号方法的数据需求

引擎必须根据策略的 `signal.method` 提供以下数据字段：

| 方法 | `clob` | `klines` | `indicators` | `trades` | `history` |
|------|--------|----------|-------------|----------|-----------|
| `clob_follow` | 必须 | - | - | - | - |
| `clob_fade` | 必须 | - | - | - | - |
| `fixed` | 必须 | - | - | - | - |
| `candle` | 必须 | 必须 | - | - | - |
| `rsi_reversal` | 必须 | 必须 | 必须 | - | - |
| `scoring` | 必须 | 必须 | 必须 | - | - |
| `ml` | 必须 | 必须 | 必须 | 必须 | 必须 |
| `post_close` | 必须 | - | - | - | - |

额外需求：
- `filter.volatility` 需要 `indicators.atrPct`
- `filter.trend` 需要 `klines`
- `entry.sizing.method = "kelly"` 需要 `history.results`
- `risk.streakRules` 需要 `state.consecutiveWins` 和 `state.consecutiveLosses`
- `risk.dailyLossLimit` 需要 `state.dailyPnl`
- `entry.maxPerHour` 需要 `state.roundsThisHour`

### 3.1 各信号方法的评估规则

对于 BIP-323 第 4 节定义的每种信号方法，引擎必须按以下规则评估方向：

#### clob_follow / clob_fade

```
令 upAsk = clob.upBestAsk
令 downAsk = clob.downBestAsk

若 upAsk 或 downAsk 不可用 → 跳过 "no_liquidity"
若 upAsk < downAsk  → 多数方向 = "up"
若 downAsk < upAsk  → 多数方向 = "down"
若 upAsk == downAsk → 跳过 "no_majority"

clob_follow → 方向 = 多数方向
clob_fade   → 方向 = 多数方向的反方向
```

卖价更低的一方（更容易买入）被视为"多数方向"——反映市场集体看好哪个结果。使用卖价而非中间价，因为卖价代表实际可执行的入场价格。

方向判定与价格过滤（`entry.minPrice`、`entry.maxPrice`）是独立关注点，在不同步骤中评估（第 7 节，步骤 8-9）。

#### candle（follow 模式）

```
令 last = klines 的最后一根
若 last.close > last.open → 蜡烛方向 = "up"
若 last.close < last.open → 蜡烛方向 = "down"
若 last.close == last.open → 跳过 "no_signal"

若 params.reverse 为真 → 方向 = 蜡烛方向的反方向
否则                   → 方向 = 蜡烛方向
```

#### candle（streak 模式）

```
令 recent = 最近 params.count 根蜡烛
令 dirs = recent 每根的方向（close > open ? "up" : "down"）
若 dirs 不全部相同 → 跳过 "no_signal"

若 params.action == "follow"  → 方向 = dirs[0]
若 params.action == "reverse" → 方向 = dirs[0] 的反方向
```

#### candle（template 模式）

```
令 n = params.sequence 长度
令 recent = 最近 n 根蜡烛
令 totalNonAny = 0, matched = 0

对每根蜡烛 i（0 到 n）：
  令 tpl = params.sequence[i]
  令 actual = classify(recent[i])
  对 [direction, bodySize, upperWick, lowerWick] 中的每个特征：
    若 tpl[特征] == "any" → 跳过（不计入）
    totalNonAny++
    若 tpl[特征] == actual[特征] → matched++

similarity = totalNonAny == 0 ? 1.0 : matched / totalNonAny
若 similarity < params.minSimilarity → 跳过 "template_no_match"

若 params.direction == "up" 或 "down" → 方向 = params.direction
若 params.direction == "reverse" → 方向 = classify(recent[n-1]).direction 的反方向
```

**classify（蜡烛分类）：**

| 特征 | 分类规则 |
|------|----------|
| direction | close > open → "bull"，close < open → "bear"，否则 → "any" |
| bodySize | 实体/振幅 > 0.7 → "large"，> 0.3 → "medium"，否则 → "small" |
| upperWick | (high - max(open,close)) / 振幅 > 0.5 → "long"，否则 → "short" |
| lowerWick | (min(open,close) - low) / 振幅 > 0.5 → "long"，否则 → "short" |

当振幅（high - low）为 0 时，bodySize = "small"，影线 = "short"。

#### rsi_reversal

```
若 indicators.rsi14 为 null → 跳过 "data_unavailable"
若 rsi14 <= params.oversold   → 方向 = "up"
若 rsi14 >= params.overbought → 方向 = "down"
否则 → 跳过 "no_signal"
```

#### scoring

```
对每个指标：
  value = compute(indicator.type, indicator.config, klines)
  若 value 为 null：
    若 indicator.required → 跳过 "signal_veto"
    否则 → 继续下一个

  对每条规则（首个命中的生效）：
    若 value 落入 rule.when 范围：
      若 rule.veto → 跳过 "signal_veto"
      scoreUp   += rule.up   ?? rule.score ?? 0
      scoreDown += rule.down ?? rule.score ?? 0
      break

若 scoreUp >= threshold 且 scoreUp >= scoreDown → 方向 = "up"
若 scoreDown >= threshold 且 scoreDown > scoreUp → 方向 = "down"
若 allowReverse 且 scoreUp <= reverseThreshold → 方向 = "down"
若 allowReverse 且 scoreDown <= reverseThreshold → 方向 = "up"
否则 → 跳过 "no_signal"
```

#### fixed

```
方向 = params.direction   // 始终入场，信号层面不会跳过
```

#### ml

```
features = computeFeatures(roundContext)   // 按第 6.2 节
prediction = model.predict(features)       // → ModelOutput
若 prediction.confidence < params.minConfidence → 跳过 "low_confidence"
方向 = prediction.direction
```

#### post_close

```
// 市场关闭后，轮询 CLOB 中间价
循环直到 params.timeoutSec 耗尽：
  若 clob.upMid >= params.confidencePrice → 方向 = "up"，退出循环
  若 clob.downMid >= params.confidencePrice → 方向 = "down"，退出循环
  等待(pollInterval)
否则 → 跳过 "no_signal"
```

### 4. SignalOutput

策略评估器在每个评估周期必须返回 SignalOutput。

```typescript
interface SignalOutput {
  action: "enter" | "skip";
  direction?: "up" | "down";          // action = "enter" 时必须
  confidence?: number;                 // 0-1，可选
  skipReason?: string;                 // action = "skip" 时必须

  // ── 诊断信息（可选，用于日志/UI）──
  diagnostics?: {
    signalMethod: string;              // 产出此决策的方法
    score?: number;                    // 打分总分（scoring 方法）
    indicators?: {                     // 各指标结果
      name: string;
      value: number | null;
      score: number;
    }[];
    modelProbUp?: number;              // ML 模型概率
    modelProbDown?: number;
    filterResults?: {                  // 各过滤器通过/未通过
      name: string;
      passed: boolean;
      reason?: string;
    }[];
  };
}
```

#### 4.1 跳过原因

实现方应该使用以下标准化的跳过原因字符串：

| 原因 | 含义 |
|------|------|
| `"no_signal"` | 信号方法未产出可操作信号（如 RSI 在中性区间） |
| `"price_above_max"` | 最优卖价 > entry.maxPrice |
| `"price_below_min"` | 最优卖价 < entry.minPrice |
| `"outside_window"` | 不在任何入场时间窗口内 |
| `"max_per_hour"` | 小时轮数限制已达 |
| `"daily_loss_limit"` | 每日亏损限额已达 |
| `"streak_pause"` | 连胜/连败规则触发暂停 |
| `"filter_schedule"` | 不在预定交易时段内 |
| `"filter_volatility"` | 波动率条件未满足 |
| `"filter_trend"` | 趋势条件未满足 |
| `"signal_veto"` | 指标否决触发 |
| `"low_confidence"` | ML 模型置信度低于阈值 |
| `"template_no_match"` | 蜡烛模板相似度低于阈值 |

### 5. ExitSignal

持仓期间，引擎在每次数据更新时评估离场规则。触发时：

```typescript
interface ExitSignal {
  action: "exit" | "hold";
  exitType?: string;                   // action = "exit" 时必须
  reason?: string;
  currentPrice?: number;
  currentProfit?: number;
  drawdownFromPeak?: number;
}
```

| exitType | 触发条件 |
|----------|----------|
| `"takeProfit"` | 利润 % >= profitPct |
| `"trailingStop"` | 从最高点回撤 >= drawdownPct |
| `"checkpoint"` | 信号重新评估后方向反转 |
| `"stopLoss"` | 价格 <= risk.stopLoss.price 且已持有 >= minHoldSec |
| `"settlement"` | 轮次结束，持仓自动结算 |

### 6. ML 模型接口

当 `signal.method = "ml"` 时，引擎必须计算特征向量并传递给模型。模型必须返回预测结果。

#### 6.1 ModelInput

```typescript
interface ModelInput {
  features: Record<string, number>;    // 命名特征映射
  metadata: {
    roundId: string;
    timestamp: number;                 // Unix 毫秒
    modelId: string;
    modelType: string;                 // "random_forest" | "xgboost" | "gbdt"
  };
}
```

#### 6.2 特征分类

特征向量按类别组织。模型的 BIP-323 `params.features` 字段（如存在）列出其使用的类别。引擎必须计算所有请求的类别。

**K 线特征**（前缀 `k_`）：

| 特征 | 计算方式 |

#### 6.3 ModelOutput

```typescript
interface ModelOutput {
  direction: "up" | "down";
  confidence: number;                  // 0.5-1.0
  probUp: number;                      // UP 概率
  probDown: number;                    // DOWN 概率
}
```

`confidence` = max(probUp, probDown)。`direction` = probUp > probDown ? "up" : "down"。

当 `confidence < params.minConfidence` 时，引擎必须跳过本轮。

#### 6.4 模型可移植性

模型文件必须自包含。除第 6.2 节定义的特征向量外，不得依赖外部状态。模型文件格式是实现特定的（JSON、ONNX、pickle 等），但第 6.1-6.3 节定义的 I/O 契约是通用的。

当模型发布到市场（BIP-323 `params.model` 使用 `market:` 前缀）时，其所需的特征类别必须与模型文件一同标注。无法计算所需特征类别的引擎必须以 `"unsupported_features"` 错误拒绝该策略。

### 7. 评估顺序

使用 RoundContext 评估 BIP-323 策略时，引擎必须遵循以下顺序：

```
 1. 检查 state.roundsThisHour < entry.maxPerHour     → 跳过 "max_per_hour"
 2. 检查 state.dailyPnl > -risk.dailyLossLimit       → 跳过 "daily_loss_limit"
 3. 检查 risk.streakRules 对 state                    → 跳过 "streak_pause"
 4. 检查 filter.schedule 对当前时间                    → 跳过 "filter_schedule"
 5. 检查 filter.volatility 对 indicators.atrPct       → 跳过 "filter_volatility"
 6. 检查 filter.trend 对 klines                       → 跳过 "filter_trend"
 7. 检查 entry.windows 对 remainingSec                → 跳过 "outside_window"
 8. 评估 signal.method → direction + confidence
 9. 检查 entry.minPrice <= bestAsk <= entry.maxPrice  → 跳过 "price_above/below"
10. 返回 SignalOutput { action: "enter", direction }
```

步骤 1-7 是预检查。任何一步失败即立即返回 `skip`，不评估信号。这确保昂贵的计算（信号打分、ML 推理）仅在所有前置条件通过时才执行。

### 8. 错误处理

| 错误 | 行为 |
|------|------|
| 需要 `klines` 但不可用 | 跳过本轮，原因 `"data_unavailable"` |
| 需要 `trades` 但不可用 | 跳过本轮，原因 `"data_unavailable"` |
| 指标计算返回 null | scoring 中跳过该指标；rsi_reversal 中跳过本轮 |
| ML 模型推理失败 | 跳过本轮，原因 `"model_error"` |
| scoring 中出现未知指标类型 | 跳过该指标，继续处理其余指标 |

错误不得导致引擎崩溃。所有错误必须通过 diagnostics 字段记录和报告。

## 设计理由

**为什么用 RoundContext 而不是原始市场数据？** 将数据预结构化为类型化的上下文，确保所有引擎提供相同的信息。否则引擎 A 可能提供 30 根 K 线而引擎 B 提供 50 根，导致不同的 RSI 值。

**为什么 ML 特征使用标准化命名？** 在一个引擎上训练的模型必须能在另一个引擎上运行。如果特征名称或计算方式不同，模型会产生垃圾结果。`k_`/`t_`/`p_`/`h_`/`time_` 前缀约定使特征自文档化。

**为什么固定评估顺序？** 廉价检查（状态、时间表）在昂贵检查（信号打分、ML 推理）之前，是所有引擎都应遵循的优化。将其设为强制性确保一致的跳过原因。

**为什么 SignalOutput 中有诊断信息？** 没有诊断信息就无法调试策略。"为什么跳过了？"需要结构化的答案，而不仅仅是"跳过了"。diagnostics 字段在生产中是可选的，但对开发来说极为宝贵。

## 安全考虑

引擎在未经用户同意的情况下，不得将 RoundContext 数据传递给不可信的外部服务。`trades` 字段包含可能具有商业敏感性的市场微观结构数据。ML 模型推理必须在本地执行；模型文件不得向外部发送数据。
