## Abstract

Defines the BiuBiu account system based on WebAuthn Passkeys, Safe smart wallet derivation, membership NFT on Arbitrum, and cross-endpoint authentication protocol.

## Motivation

The BiuBiu ecosystem spans multiple independent endpoints (spaces). Users need a single identity that works across all endpoints without trusting any individual endpoint with their private keys. Passkeys provide phishing-resistant, device-bound authentication. Safe wallets provide programmable on-chain accounts. This BIP combines them into a unified identity layer.

## Specification

### 1. Account Creation

A BiuBiu account is created on `biubiu.tools` via WebAuthn Passkey registration.

```
User clicks "Create Account"
  → Browser triggers navigator.credentials.create()
  → Device generates P-256 keypair (ECDSA on secp256r1)
  → Public key registered on biubiu.tools
  → biubiu.tools derives a Safe wallet address from the public key
  → Account is ready
```

The account consists of:

| Component | Storage | Purpose |
|-----------|---------|---------|
| Passkey private key | Device secure enclave (never leaves device) | Authentication signing |
| Passkey public key | biubiu.tools server | Identity verification |
| Safe wallet address | On-chain (Arbitrum) | On-chain identity, asset ownership |
| Profile | biubiu.tools server | Display name, avatar, public profile URL |

### 2. Safe Wallet Derivation

The user's on-chain wallet is a Safe (formerly Gnosis Safe) smart contract wallet on Arbitrum, deterministically derived from the passkey public key.

```
safeAddress = CREATE2(
  factory:  SafeProxyFactory on Arbitrum,
  salt:     keccak256(passkeyPublicKey),
  initCode: Safe singleton + setup(owners: [passkeyVerifier], threshold: 1)
)
```

The passkey signs transactions via a **PasskeyVerifier** contract that validates WebAuthn signatures (P-256 / secp256r1) on-chain. This follows EIP-7212 (secp256r1 precompile) where available, or uses a Solidity verifier fallback.

**Properties:**
- Same passkey always derives the same Safe address (deterministic)
- Safe is deployed lazily (on first on-chain transaction, not at registration)
- Passkey is the sole owner at creation time
- User can add additional owners later (recovery keys, hardware wallets)

### 3. Profile

Each account has a public profile accessible at:

```
https://biubiu.tools/profile/{userId}
```

Profile data:

| Field | Type | Public | Description |
|-------|------|--------|-------------|
| `userId` | string | yes | Unique identifier (e.g., `user_abc123`) |
| `displayName` | string | yes | Chosen display name |
| `avatar` | string | yes | Avatar URL |
| `safeAddress` | address | yes | Arbitrum Safe wallet address |
| `passkeyPublicKey` | bytes | yes | P-256 public key (for signature verification) |
| `createdAt` | string | yes | ISO 8601 |

The profile URL is used by endpoint admins to add users to spaces (see Section 5).

### 4. Membership

Membership is represented by an ERC-721 NFT on Arbitrum.

```solidity
interface IBiubiuMembership {
    /// @notice Mint membership by paying subscription fee
    /// @param months 1 = monthly ($100), 12 = annual ($1000)
    function subscribe(uint8 months) external payable;

    /// @notice Check if user has active membership
    function isActive(address user) external view returns (bool);

    /// @notice Get membership expiry
    function expiresAt(uint256 tokenId) external view returns (uint256);

    /// @notice Extend existing membership
    function extend(uint256 tokenId, uint8 months) external payable;
}
```

**Properties:**
- NFT is minted to the user's Safe address
- NFT is **transferable** (can sell or gift membership)
- Membership grants: 100 strategy instances (vs 3 for free), live trading access, full analytics
- Price: $100/month or $1000/year, paid in ETH on Arbitrum (Chainlink oracle for conversion)
- When NFT is transferred, membership benefits transfer with it

### 5. Cross-Endpoint Authentication

How a BiuBiu user logs into any third-party endpoint:

```
Step 1: User visits endpoint (e.g., btc-bot.alpha-club.xyz)

Step 2: Endpoint presents a challenge
  → GET /api/v1/auth/challenge
  → { challenge: "<random-32-bytes-hex>", expiresAt: "<ISO>" }

Step 3: User signs the challenge with passkey
  → Browser: navigator.credentials.get()
  → Produces: { authenticatorData, clientDataJSON, signature }

Step 4: User submits signed challenge to endpoint
  → POST /api/v1/auth/login
  → Body: {
      userId: "user_abc123",
      challenge: "<from step 2>",
      authenticatorData: "<bytes>",
      clientDataJSON: "<bytes>",
      signature: "<bytes>"
    }

Step 5: Endpoint verifies signature
  → Fetch public key: GET https://biubiu.tools/api/identity/{userId}/public-key
  → Verify WebAuthn signature against public key + challenge
  → If valid: create session, return token

Step 6: Authenticated
  → Response: {
      token: "<session-token>",
      expiresAt: "<ISO>",
      userId: "user_abc123",
      safeAddress: "0x...",
      role: "member",         // Assigned by endpoint admin
      permissions: { ... }
    }
```

**Security properties:**
- Endpoint NEVER sees the passkey private key
- Endpoint verifies against biubiu.tools public key registry (trusted source)
- Challenge is single-use and time-limited (5 minutes)
- Session token is endpoint-specific (not a universal token)

### 6. Public Key Registry

biubiu.tools maintains a public key registry that endpoints use for verification:

```
GET https://biubiu.tools/api/identity/{userId}/public-key
→ {
    userId: "user_abc123",
    publicKey: "<P-256 public key, base64url>",
    algorithm: "ES256",
    safeAddress: "0x...",
    createdAt: "<ISO>"
  }
```

Endpoints SHOULD cache this response (public keys don't change). Cache TTL: 24 hours.

Endpoints MUST verify the response is from biubiu.tools (TLS certificate pinning or response signing in future versions).

### 7. Account Recovery

If a user loses their device (and thus their passkey):

1. **Additional passkey:** User can register multiple passkeys (e.g., phone + laptop). Any one can authenticate.
2. **Recovery key:** User can add an EOA or another Safe as a recovery owner of their Safe wallet.
3. **Social recovery:** Future — trusted contacts can collectively restore access.

The Safe wallet's multi-owner model enables flexible recovery without a single point of failure.

## Rationale

**Why Passkeys over seed phrases?** Passkeys are device-bound, phishing-resistant, and familiar to non-crypto users ("just use your fingerprint"). Seed phrases are error-prone and regularly stolen via phishing.

**Why Safe over EOA?** Safe wallets support multiple signers, key rotation, spending limits, and account recovery. EOAs are a single private key — lose it, lose everything. Safe also enables the sub-account delegation model (BIP-327).

**Why Arbitrum?** Low gas fees make membership NFT operations affordable (<$0.10 per transaction). Ethereum L1 would cost $5-50 per membership action.

## Security Considerations

- Passkey private keys MUST remain in the device's secure enclave. biubiu.tools MUST NOT request or store private keys.
- The public key registry is a centralized trust point. If biubiu.tools is compromised, attackers could register fake public keys. Mitigation: on-chain public key registry in future versions.
- Endpoints MUST validate challenges they issued (prevent replay attacks from other endpoints).
