## 摘要

定义基于 WebAuthn Passkey 的 BiuBiu 账号系统，包括 Safe 智能钱包派生、Arbitrum 上的会员 NFT、以及跨端点认证协议。

## 动机

BiuBiu 生态跨越多个独立端点（空间）。用户需要一个在所有端点通用的身份，且不需要将私钥信任给任何端点。Passkey 提供防钓鱼的、设备绑定的认证。Safe 钱包提供可编程的链上账户。本 BIP 将二者统一为身份层。

## 规范

### 1. 账号创建

BiuBiu 账号在 `biubiu.tools` 上通过 WebAuthn Passkey 注册创建。

```
用户点击"创建账号"
  → 浏览器触发 navigator.credentials.create()
  → 设备生成 P-256 密钥对（secp256r1 上的 ECDSA）
  → 公钥注册到 biubiu.tools
  → biubiu.tools 从公钥派生 Safe 钱包地址
  → 账号就绪
```

账号组成：

| 组件 | 存储位置 | 用途 |
|------|----------|------|
| Passkey 私钥 | 设备安全芯片（永不离开设备）| 认证签名 |
| Passkey 公钥 | biubiu.tools 服务器 | 身份验证 |
| Safe 钱包地址 | 链上（Arbitrum）| 链上身份、资产持有 |
| 个人资料 | biubiu.tools 服务器 | 显示名、头像、公开资料 URL |

### 2. Safe 钱包派生

用户的链上钱包是 Arbitrum 上的 Safe 智能合约钱包，从 passkey 公钥确定性派生。

```
safeAddress = CREATE2(
  factory:  Arbitrum 上的 SafeProxyFactory,
  salt:     keccak256(passkeyPublicKey),
  initCode: Safe singleton + setup(owners: [passkeyVerifier], threshold: 1)
)
```

Passkey 通过 **PasskeyVerifier** 合约签署交易，该合约在链上验证 WebAuthn 签名（P-256 / secp256r1）。支持 EIP-7212（secp256r1 预编译）或 Solidity 验证器回退。

**特性：**
- 相同 passkey 总是派生相同的 Safe 地址（确定性）
- Safe 延迟部署（首次链上交易时部署，非注册时）
- 创建时 passkey 是唯一所有者
- 用户可后续添加额外所有者（恢复密钥、硬件钱包）

### 3. 个人资料

每个账号有公开资料页：

```
https://biubiu.tools/profile/{userId}
```

| 字段 | 类型 | 是否公开 | 描述 |
|------|------|----------|------|
| `userId` | string | 是 | 唯一标识（如 `user_abc123`）|
| `displayName` | string | 是 | 显示名称 |
| `avatar` | string | 是 | 头像 URL |
| `safeAddress` | address | 是 | Arbitrum Safe 钱包地址 |
| `passkeyPublicKey` | bytes | 是 | P-256 公钥（用于签名验证）|
| `createdAt` | string | 是 | ISO 8601 |

端点管理员通过资料 URL 将用户添加到空间。

### 4. 会员

会员以 Arbitrum 上的 ERC-721 NFT 表示。

```solidity
interface IBiubiuMembership {
    /// @notice 支付订阅费铸造会员
    /// @param months 1 = 月付 ($100)，12 = 年付 ($1000)
    function subscribe(uint8 months) external payable;

    /// @notice 检查用户是否有有效会员
    function isActive(address user) external view returns (bool);

    /// @notice 获取会员到期时间
    function expiresAt(uint256 tokenId) external view returns (uint256);

    /// @notice 延长现有会员
    function extend(uint256 tokenId, uint8 months) external payable;
}
```

**特性：**
- NFT 铸造到用户的 Safe 地址
- NFT **可转让**（可以出售或赠送会员资格）
- 会员权益：100 个策略实例（免费用户 3 个）、实盘交易权限、完整分析功能
- 价格：$100/月 或 $1000/年，以 ETH 在 Arbitrum 上支付（Chainlink 预言机换算）
- NFT 转让时，会员权益随之转移

### 5. 跨端点认证

BiuBiu 用户如何登录任何第三方端点：

```
步骤 1：用户访问端点（如 btc-bot.alpha-club.xyz）

步骤 2：端点出示挑战
  → GET /api/v1/auth/challenge
  → { challenge: "<随机32字节hex>", expiresAt: "<ISO>" }

步骤 3：用户用 passkey 签署挑战
  → 浏览器：navigator.credentials.get()
  → 产出：{ authenticatorData, clientDataJSON, signature }

步骤 4：用户提交签名挑战到端点
  → POST /api/v1/auth/login
  → Body: {
      userId: "user_abc123",
      challenge: "<来自步骤 2>",
      authenticatorData: "<bytes>",
      clientDataJSON: "<bytes>",
      signature: "<bytes>"
    }

步骤 5：端点验证签名
  → 获取公钥：GET https://biubiu.tools/api/identity/{userId}/public-key
  → 验证 WebAuthn 签名：公钥 + 挑战
  → 验证通过：创建会话，返回 token

步骤 6：认证完成
  → 响应：{
      token: "<会话token>",
      expiresAt: "<ISO>",
      userId: "user_abc123",
      safeAddress: "0x...",
      role: "member",
      permissions: { ... }
    }
```

**安全特性：**
- 端点永远看不到 passkey 私钥
- 端点通过 biubiu.tools 公钥注册表验证（可信来源）
- 挑战是一次性的、有时间限制的（5 分钟）
- 会话 token 是端点特定的（不是全局 token）

### 6. 公钥注册表

biubiu.tools 维护公钥注册表，端点用于验证：

```
GET https://biubiu.tools/api/identity/{userId}/public-key
→ {
    userId: "user_abc123",
    publicKey: "<P-256 公钥, base64url>",
    algorithm: "ES256",
    safeAddress: "0x...",
    createdAt: "<ISO>"
  }
```

端点应该缓存此响应（公钥不会变）。缓存 TTL：24 小时。

### 7. 账号恢复

用户丢失设备（从而丢失 passkey）时：

1. **额外 passkey：** 用户可注册多个 passkey（如手机 + 笔记本）。任一个都可认证。
2. **恢复密钥：** 用户可添加 EOA 或另一个 Safe 作为恢复所有者。
3. **社交恢复：** 未来——可信联系人可集体恢复访问权。

Safe 钱包的多所有者模型实现了灵活恢复，没有单点故障。

## 设计理由

**为什么用 Passkey 而不是助记词？** Passkey 绑定设备、防钓鱼，非加密用户也熟悉（"用指纹就行"）。助记词容易出错，经常被钓鱼窃取。

**为什么用 Safe 而不是 EOA？** Safe 支持多签名者、密钥轮换、支出限额和账号恢复。EOA 是单个私钥——丢了就全没了。Safe 还为小号委托模型（BIP-327）提供基础。

**为什么选 Arbitrum？** 低 gas 费使会员 NFT 操作可负担（<$0.10/交易）。以太坊 L1 每次操作需 $5-50。

## 安全考虑

- Passkey 私钥必须留在设备安全芯片中。biubiu.tools 不得请求或存储私钥。
- 公钥注册表是中心化信任点。若 biubiu.tools 被入侵，攻击者可注册假公钥。缓解措施：未来版本使用链上公钥注册表。
- 端点必须验证是自己签发的挑战（防止其他端点的重放攻击）。
