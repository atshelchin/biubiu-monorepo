## Abstract

Defines the sub-account (小号) derivation, EOA signer delegation, risk isolation model, and security requirements for automated strategy execution across BiuBiu endpoints.

## Motivation

Automated strategy execution requires frequent signing (every round, every 5 minutes). Passkey signing requires user interaction (biometric prompt) — impractical for 24/7 automation. Sub-accounts with delegated EOA signers solve this: the endpoint holds a signing key that can act on behalf of a restricted sub-account, without access to the user's main account or assets on other endpoints.

## Specification

### 1. Sub-Account Derivation

A sub-account is a Safe wallet derived from three inputs:

```
subAccountAddress = CREATE2(
  factory:  BiuBiuSubAccountFactory on Arbitrum,
  salt:     keccak256(passkeyPublicKey, endpointDomain, nonce),
  initCode: Safe singleton + setup(owners: [passkeyVerifier], threshold: 1)
)
```

| Input | Type | Purpose |
|-------|------|---------|
| `passkeyPublicKey` | bytes | The user's BIP-326 passkey public key |
| `endpointDomain` | string | The endpoint's domain (e.g., `btc-bot.alpha-club.xyz`) |
| `nonce` | uint256 | Sub-account index (0, 1, 2, ...) — allows multiple sub-accounts per endpoint |

**Key properties:**

1. **Deterministic:** Same inputs always produce the same address. User and endpoint can independently compute the address.
2. **Endpoint-isolated:** Different `endpointDomain` → different sub-account address. Assets and risk are siloed per endpoint.
3. **Multi-account:** Incrementing `nonce` creates additional sub-accounts on the same endpoint (e.g., one for sandbox, one for live).
4. **Lazily deployed:** The Safe contract is deployed on first use, not at derivation time.

### 2. Sub-Account Lifecycle

```
Step 1: User decides to create a sub-account on an endpoint

Step 2: User signs a SubAccountCreation message with passkey
  → Message: {
      action: "create_sub_account",
      endpointDomain: "btc-bot.alpha-club.xyz",
      nonce: 0,
      timestamp: "<ISO>"
    }
  → Signature proves: "I (passkey holder) authorize this sub-account"

Step 3: Endpoint computes sub-account address
  → subAccountAddress = derive(passkeyPubKey, domain, nonce)

Step 4: Endpoint generates an EOA keypair for automation
  → { eoaAddress, eoaPrivateKey } = generateKeypair()
  → Endpoint stores eoaPrivateKey securely (encrypted, HSM, etc.)

Step 5: User adds EOA signer to sub-account
  → User signs an addOwner transaction on the sub-account Safe:
    safe.addOwnerWithThreshold(eoaAddress, threshold: 1)
  → This is a passkey-signed on-chain transaction
  → ⚠️ RISK WARNING displayed (see Section 6)

Step 6: Sub-account is ready for automation
  → Endpoint can now sign transactions using eoaPrivateKey
  → These transactions execute from subAccountAddress
```

### 3. EOA Signer Delegation

After Step 5, the sub-account Safe has two owners:

```
Sub-Account Safe (on Arbitrum)
├── Owner 1: PasskeyVerifier    ← User's passkey (full control)
└── Owner 2: EOA Signer         ← Endpoint's automation key
    Threshold: 1                 ← Either owner can sign
```

**What the EOA signer CAN do:**
- Sign strategy trade transactions (buy/sell prediction market tokens)
- Execute within the sub-account's balance
- Only operate on the sub-account's assets

**What the EOA signer CANNOT do:**
- Access the user's main Safe wallet (BIP-326)
- Access sub-accounts on other endpoints
- Move assets to arbitrary addresses (if spending limits are configured)
- Modify the Safe's owner list (requires passkey signature)

### 4. Cross-Endpoint Isolation

```
User: shelchin (passkey: 0xABC...)

Main Account (BIP-326):
  Safe: 0x1111...
  Assets: Membership NFT, personal funds
  Signer: Passkey only

Sub-Account on alpha-club.xyz (nonce=0):
  Safe: 0x2222...  = derive(0xABC, "btc-bot.alpha-club.xyz", 0)
  Assets: $500 trading capital
  Signers: Passkey + EOA(0xAAA)   ← alpha-club holds 0xAAA private key

Sub-Account on my-bot.example.com (nonce=0):
  Safe: 0x3333...  = derive(0xABC, "my-bot.example.com", 0)
  Assets: $200 trading capital
  Signers: Passkey + EOA(0xBBB)   ← my-bot holds 0xBBB private key

Compromise scenario:
  alpha-club.xyz's EOA key (0xAAA) is leaked
  → Attacker can drain 0x2222 ($500)
  → Attacker CANNOT touch 0x3333 ($200) — different key
  → Attacker CANNOT touch 0x1111 (main) — no access
  → Damage is contained to one endpoint
```

### 5. Endpoint Integration

How an endpoint provides sub-account services:

#### 5.1 Sub-Account Discovery

```
POST /api/v1/sub-account/derive
Body: {
  userId: "user_abc123",
  nonce: 0
}
→ {
    subAccountAddress: "0x2222...",
    deployed: false,
    eoaSigner: null              // Not yet created
  }
```

#### 5.2 Create & Activate

```
POST /api/v1/sub-account/activate
Body: {
  userId: "user_abc123",
  nonce: 0,
  signature: "<passkey signature of activation message>"
}
→ {
    subAccountAddress: "0x2222...",
    eoaSigner: "0xAAA...",       // Endpoint's generated EOA address
    deployed: true,
    // User must now call safe.addOwnerWithThreshold(0xAAA, 1)
    // to complete setup
    addOwnerTx: {                // Pre-built transaction for user to sign
      to: "0x2222...",
      data: "<encoded addOwner call>",
      value: 0
    }
  }
```

#### 5.3 Status

```
GET /api/v1/sub-account/status?userId={userId}&nonce={nonce}
→ {
    subAccountAddress: "0x2222...",
    deployed: true,
    eoaSigner: "0xAAA...",
    eoaSignerActive: true,       // true if EOA is an owner of the Safe
    balance: "0.5",              // ETH balance
    tokenBalances: [             // Prediction market token balances
      { token: "UP-BTC-...", balance: "100" },
      { token: "DOWN-BTC-...", balance: "50" }
    ]
  }
```

#### 5.4 Remove EOA Signer (User-initiated)

User can revoke the endpoint's signing authority at any time:

```
POST /api/v1/sub-account/revoke-signer
Body: {
  userId: "user_abc123",
  nonce: 0,
  signature: "<passkey signature>"
}
```

This submits a `removeOwner` transaction on the Safe, signed by the user's passkey. The endpoint's EOA is removed. Automation stops immediately.

### 6. Risk Warnings

Implementations MUST display clear risk warnings at these points:

#### 6.1 When creating a sub-account for live trading

```
⚠️ Live Trading Sub-Account

You are creating a sub-account on [endpoint-name].
This endpoint will receive a signing key that can execute
trades using funds in this sub-account.

• Only deposit what you can afford to lose
• The endpoint operator controls the signing key
• If the endpoint is compromised, funds in this sub-account may be lost
• Your main BiuBiu account and other endpoints are NOT affected

Sub-account address: 0x2222...
Endpoint: btc-bot.alpha-club.xyz

[Cancel]  [I understand the risks, proceed]
```

#### 6.2 When adding the EOA signer

```
⚠️ Adding Automation Signer

You are granting [endpoint-name] permission to sign transactions
on your sub-account without requiring your passkey each time.

This means the endpoint can:
  ✓ Execute strategy trades automatically
  ✓ Buy and sell prediction market tokens
  ✗ Cannot access your main account
  ✗ Cannot access other endpoints' sub-accounts

You can revoke this permission at any time.

EOA Signer: 0xAAA...
Sub-account: 0x2222...

[Cancel]  [Add signer]
```

#### 6.3 When depositing funds

```
⚠️ Depositing to Sub-Account

You are depositing [amount] ETH to your sub-account on [endpoint-name].
This endpoint's automation key can use these funds for trading.

Recommended: Start with a small amount and increase after testing.

[Cancel]  [Deposit]
```

### 7. EOA Signer Security Requirements

Endpoint operators MUST follow these security requirements for EOA private key management:

| Requirement | Level | Description |
|-------------|-------|-------------|
| Key encryption at rest | MUST | Private keys encrypted with AES-256-GCM or equivalent |
| Key isolation | MUST | Each sub-account's key stored separately, not in a shared keystore |
| Access logging | MUST | All signing operations logged with timestamp, transaction hash, sub-account |
| Key rotation | SHOULD | Support re-keying (generate new EOA, user swaps owners) |
| HSM support | SHOULD | Hardware security module for key storage where available |
| Backup policy | MUST | Encrypted backups with documented recovery procedure |

Endpoint operators MUST NOT:
- Store private keys in plaintext
- Share private keys across sub-accounts
- Use a single master key for all users
- Log private keys in any form

### 8. Sub-Account Spending Limits (Optional)

For additional safety, users MAY configure spending limits on the sub-account Safe using Safe's native module system:

```solidity
// Allowance module: limits how much the EOA signer can spend per period
safe.enableModule(allowanceModule);
allowanceModule.setAllowance(
  eoaSigner,          // delegate
  tokenAddress,       // which token
  100e6,             // max amount per period (e.g., 100 USDC)
  60 * 60 * 24,      // reset period (24 hours)
  0                   // reset base time
);
```

This limits the blast radius even if the EOA key is fully compromised: attacker can spend at most $100/day.

## Rationale

**Why derive sub-accounts from domain name?** Binding to `endpointDomain` ensures that an endpoint cannot compute or access sub-accounts meant for other endpoints. Even if two endpoints collude, they cannot merge or cross-access sub-accounts.

**Why Safe over a simple EOA sub-account?** Safe allows the user to always retain passkey-based override authority. With a plain EOA, whoever holds the private key has full control — there's no override mechanism. Safe's multi-owner model means the user's passkey can always remove the EOA signer, even if the endpoint goes rogue.

**Why threshold 1 instead of 2?** Threshold 1 means either owner can sign independently. This is required for automation (endpoint signs without user). The user's passkey provides override/recovery capability, not co-signing. If threshold were 2, every trade would need both signatures, defeating the purpose.

**Why not use session keys or EIP-7715?** Session keys are a lighter-weight alternative but require specific smart contract support. Safe's owner model is battle-tested, widely deployed, and understood. Future BIPs may adopt session keys when the tooling matures.

## Security Considerations

- **Endpoint trust:** Users MUST understand that creating a sub-account on an endpoint trusts that endpoint with the sub-account's funds. The risk warnings in Section 6 are REQUIRED, not optional.
- **Domain binding:** The `endpointDomain` in the derivation MUST be the actual domain serving the endpoint API, not a user-chosen label. This prevents impersonation.
- **Main account isolation:** The main Safe (BIP-326) MUST NEVER have an endpoint's EOA as an owner. Only sub-accounts have delegated signers.
- **Nonce management:** The `nonce` is chosen by the user. Endpoints MUST NOT auto-increment nonces. Users should be aware of which nonces they've used.
