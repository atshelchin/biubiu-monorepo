## 摘要

定义小号派生、EOA 签名者委托、风险隔离模型，以及跨 BiuBiu 端点自动化策略执行的安全要求。

## 动机

自动化策略执行需要频繁签名（每轮、每 5 分钟）。Passkey 签名需要用户交互（生物识别提示）——对 24/7 自动化不现实。小号加委托 EOA 签名者解决这个问题：端点持有一个签名密钥，可以代表受限的小号执行操作，但无法访问用户主账号或其他端点上的资产。

## 规范

### 1. 小号派生

小号是从三个输入派生的 Safe 钱包：

```
subAccountAddress = CREATE2(
  factory:  Arbitrum 上的 BiuBiuSubAccountFactory,
  salt:     keccak256(passkeyPublicKey, endpointDomain, nonce),
  initCode: Safe singleton + setup(owners: [passkeyVerifier], threshold: 1)
)
```

| 输入 | 类型 | 用途 |
|------|------|------|
| `passkeyPublicKey` | bytes | 用户的 BIP-326 passkey 公钥 |
| `endpointDomain` | string | 端点域名（如 `btc-bot.alpha-club.xyz`）|
| `nonce` | uint256 | 小号序号（0, 1, 2, ...）——允许在同一端点上创建多个小号 |

**关键特性：**

1. **确定性：** 相同输入总是产出相同地址。用户和端点可独立计算地址。
2. **端点隔离：** 不同 `endpointDomain` → 不同小号地址。资产和风险按端点隔离。
3. **多账号：** 递增 `nonce` 在同一端点创建额外小号（如一个沙盘、一个实盘）。
4. **延迟部署：** Safe 合约在首次使用时部署，非派生时。

### 2. 小号生命周期

```
步骤 1：用户决定在某端点创建小号

步骤 2：用户用 passkey 签署创建消息
  → 消息：{
      action: "create_sub_account",
      endpointDomain: "btc-bot.alpha-club.xyz",
      nonce: 0,
      timestamp: "<ISO>"
    }
  → 签名证明："我（passkey 持有者）授权此小号"

步骤 3：端点计算小号地址
  → subAccountAddress = derive(passkeyPubKey, domain, nonce)

步骤 4：端点为自动化生成 EOA 密钥对
  → { eoaAddress, eoaPrivateKey } = generateKeypair()
  → 端点安全存储 eoaPrivateKey（加密、HSM 等）

步骤 5：用户将 EOA 签名者添加到小号
  → 用户签署小号 Safe 上的 addOwner 交易：
    safe.addOwnerWithThreshold(eoaAddress, threshold: 1)
  → 这是一笔 passkey 签署的链上交易
  → ⚠️ 显示风险警告（见第 6 节）

步骤 6：小号准备就绪
  → 端点现在可以用 eoaPrivateKey 签署交易
  → 这些交易从 subAccountAddress 执行
```

### 3. EOA 签名者委托

步骤 5 之后，小号 Safe 有两个所有者：

```
小号 Safe（Arbitrum 上）
├── 所有者 1：PasskeyVerifier    ← 用户的 passkey（完全控制）
└── 所有者 2：EOA 签名者          ← 端点的自动化密钥
    阈值：1                       ← 任一所有者都可签名
```

**EOA 签名者能做什么：**
- 签署策略交易（买卖预测市场代币）
- 在小号余额范围内执行
- 仅操作小号的资产

**EOA 签名者不能做什么：**
- 访问用户的主 Safe 钱包（BIP-326）
- 访问其他端点的小号
- 向任意地址转移资产（如配置了支出限额）
- 修改 Safe 的所有者列表（需要 passkey 签名）

### 4. 跨端点隔离

```
用户：shelchin（passkey：0xABC...）

主账号（BIP-326）：
  Safe：0x1111...
  资产：会员 NFT、个人资金
  签名者：仅 Passkey

alpha-club.xyz 上的小号（nonce=0）：
  Safe：0x2222... = derive(0xABC, "btc-bot.alpha-club.xyz", 0)
  资产：$500 交易资金
  签名者：Passkey + EOA(0xAAA)   ← alpha-club 持有 0xAAA 私钥

my-bot.example.com 上的小号（nonce=0）：
  Safe：0x3333... = derive(0xABC, "my-bot.example.com", 0)
  资产：$200 交易资金
  签名者：Passkey + EOA(0xBBB)   ← my-bot 持有 0xBBB 私钥

入侵场景：
  alpha-club.xyz 的 EOA 密钥（0xAAA）泄漏
  → 攻击者可提取 0x2222（$500）
  → 攻击者无法触及 0x3333（$200）— 不同密钥
  → 攻击者无法触及 0x1111（主账号）— 无访问权
  → 损失被限制在一个端点内
```

### 5. 端点集成

端点如何提供小号服务：

#### 5.1 小号发现

```
POST /api/v1/sub-account/derive
Body: { userId: "user_abc123", nonce: 0 }
→ {
    subAccountAddress: "0x2222...",
    deployed: false,
    eoaSigner: null
  }
```

#### 5.2 创建并激活

```
POST /api/v1/sub-account/activate
Body: {
  userId: "user_abc123",
  nonce: 0,
  signature: "<passkey 签名的激活消息>"
}
→ {
    subAccountAddress: "0x2222...",
    eoaSigner: "0xAAA...",
    deployed: true,
    addOwnerTx: {                // 预构建的交易，待用户签名
      to: "0x2222...",
      data: "<编码的 addOwner 调用>",
      value: 0
    }
  }
```

#### 5.3 状态查询

```
GET /api/v1/sub-account/status?userId={userId}&nonce={nonce}
→ {
    subAccountAddress: "0x2222...",
    deployed: true,
    eoaSigner: "0xAAA...",
    eoaSignerActive: true,
    balance: "0.5",
    tokenBalances: [
      { token: "UP-BTC-...", balance: "100" },
      { token: "DOWN-BTC-...", balance: "50" }
    ]
  }
```

#### 5.4 移除 EOA 签名者（用户发起）

用户可随时撤销端点的签名授权：

```
POST /api/v1/sub-account/revoke-signer
Body: {
  userId: "user_abc123",
  nonce: 0,
  signature: "<passkey 签名>"
}
```

在 Safe 上提交 `removeOwner` 交易，由用户 passkey 签署。端点的 EOA 被移除，自动化立即停止。

### 6. 风险警告

实现方必须在以下时刻显示清晰的风险警告：

#### 6.1 创建实盘小号时

```
⚠️ 实盘交易小号

你正在 [端点名称] 上创建小号。
该端点将获得一个签名密钥，可以使用此小号中的资金执行交易。

• 仅存入你可以承受损失的金额
• 端点运营者控制签名密钥
• 如果端点被入侵，此小号中的资金可能丢失
• 你的 BiuBiu 主账号和其他端点不受影响

小号地址：0x2222...
端点：btc-bot.alpha-club.xyz

[取消]  [我理解风险，继续]
```

#### 6.2 添加 EOA 签名者时

```
⚠️ 添加自动化签名者

你正在授权 [端点名称] 在不需要你的 passkey 的情况下
代你的小号签署交易。

这意味着端点可以：
  ✓ 自动执行策略交易
  ✓ 买卖预测市场代币
  ✗ 无法访问你的主账号
  ✗ 无法访问其他端点的小号

你可以随时撤销此权限。

EOA 签名者：0xAAA...
小号：0x2222...

[取消]  [添加签名者]
```

#### 6.3 存入资金时

```
⚠️ 向小号存款

你正在向 [端点名称] 上的小号存入 [金额] ETH。
该端点的自动化密钥可以使用这些资金进行交易。

建议：先从小额开始，测试后再增加。

[取消]  [存入]
```

### 7. EOA 签名者安全要求

端点运营者必须遵循以下 EOA 私钥管理安全要求：

| 要求 | 级别 | 描述 |
|------|------|------|
| 静态加密 | 必须 | 私钥用 AES-256-GCM 或同等方式加密 |
| 密钥隔离 | 必须 | 每个小号的密钥单独存储，不共用密钥库 |
| 访问日志 | 必须 | 所有签名操作记录时间戳、交易哈希、小号 |
| 密钥轮换 | 应该 | 支持重新生成密钥（生成新 EOA，用户替换所有者）|
| HSM 支持 | 应该 | 可用时使用硬件安全模块存储密钥 |
| 备份策略 | 必须 | 加密备份，有文档化的恢复流程 |

端点运营者不得：
- 明文存储私钥
- 在小号间共享私钥
- 对所有用户使用单个主密钥
- 以任何形式记录私钥

### 8. 小号支出限额（可选）

为额外安全，用户可以使用 Safe 原生模块系统配置支出限额：

```solidity
safe.enableModule(allowanceModule);
allowanceModule.setAllowance(
  eoaSigner,          // 委托者
  tokenAddress,       // 哪个代币
  100e6,             // 每周期最大金额（如 100 USDC）
  60 * 60 * 24,      // 重置周期（24 小时）
  0                   // 重置基准时间
);
```

即使 EOA 密钥完全泄漏，攻击者每天最多只能花 $100。

## 设计理由

**为什么从域名派生小号？** 绑定 `endpointDomain` 确保端点无法计算或访问其他端点的小号。即使两个端点串通，也无法合并或交叉访问小号。

**为什么用 Safe 而不是简单的 EOA 小号？** Safe 让用户始终保留基于 passkey 的覆盖权限。用普通 EOA，谁持有私钥谁就完全控制——没有覆盖机制。Safe 的多所有者模型意味着用户的 passkey 始终可以移除 EOA 签名者，即使端点作恶。

**为什么阈值是 1 而不是 2？** 阈值 1 意味着任一所有者可独立签名。自动化需要这个（端点不需要用户参与即可签名）。用户的 passkey 提供覆盖/恢复能力，而非共签。阈值 2 的话每笔交易都需要双方签名，违背了自动化的目的。

**为什么不用 session keys 或 EIP-7715？** Session keys 是更轻量的替代方案，但需要特定的智能合约支持。Safe 的所有者模型经过实战检验、广泛部署且被理解。未来 BIP 可能在工具成熟后采用 session keys。

## 安全考虑

- **端点信任：** 用户必须理解，在端点上创建小号意味着信任该端点处理小号资金。第 6 节的风险警告是必须的，不是可选的。
- **域名绑定：** 派生中的 `endpointDomain` 必须是实际服务端点 API 的域名，不是用户自定义的标签。防止冒充。
- **主账号隔离：** 主 Safe（BIP-326）永远不得将端点的 EOA 作为所有者。只有小号有委托签名者。
- **Nonce 管理：** `nonce` 由用户选择。端点不得自动递增 nonce。用户应该知道自己用了哪些 nonce。
