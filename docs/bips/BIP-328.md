## Abstract

Defines the HTTP API that BiuBiu-compatible endpoints MUST implement. Organized into five domains: Identity, Strategy, Data, Marketplace, and Operations. Maintains backwards compatibility with existing `/api/*` endpoints while introducing `/api/v2/*` for new capabilities.

## Motivation

Third-party endpoints need a clear, complete API contract to integrate with the BiuBiu ecosystem. This BIP unifies the existing ad-hoc API (documented in the btc-updown-5m-bot API.md) with new requirements from BIP-323 (strategy format), BIP-324 (runtime interface), BIP-326 (identity), and BIP-327 (sub-accounts), plus the web UI needs identified through the prototype.

## Specification

### 0. Conventions

**Base URL:** `https://{endpoint-domain}/api/v2/`

**Authentication levels:**

| Level | Header | Description |
|-------|--------|-------------|
| `public` | none | No auth required |
| `session` | `Authorization: Bearer {token}` | Logged-in user (from BIP-326 auth flow) |
| `signed` | `Authorization: Bearer {token}` + `X-Passkey-Signature: {sig}` | Passkey-signed write operations |
| `admin` | `Authorization: Bearer {token}` + admin role | Space administrator |

**Response format:**

```json
{ "data": { ... } }                   // Success
{ "error": { "code": "...", "message": "..." } }  // Error
```

**Pagination:** `?page=1&pageSize=20` → response includes `{ total, page, pageSize }`

**Filtering:** ISO 8601 dates via `?from=&to=`, timezone via `?tz_offset=` (minutes)

**CORS:** `Access-Control-Allow-Origin: *` or configured origin whitelist.

### 1. Identity (`/api/v2/auth/*`)

Implements BIP-326 cross-endpoint authentication and BIP-327 sub-account management.

#### 1.1 Authentication

```
GET  /api/v2/auth/challenge                              [public]
```
Returns a single-use challenge for passkey signing. Expires in 5 minutes.
```json
→ { "challenge": "<hex>", "expiresAt": "<ISO>" }
```

```
POST /api/v2/auth/login                                  [public]
```
Verify passkey signature, create session.
```json
← { "userId", "challenge", "authenticatorData", "clientDataJSON", "signature" }
→ { "token", "expiresAt", "userId", "safeAddress", "role", "permissions" }
```
Engine fetches public key from `biubiu.tools/api/identity/{userId}/public-key` to verify.

```
POST /api/v2/auth/logout                                 [session]
→ { "ok": true }
```

```
GET  /api/v2/auth/me                                     [session]
→ { "userId", "safeAddress", "role", "permissions", "membership" }
```

#### 1.2 Member Management (Admin)

```
GET    /api/v2/auth/members                              [admin]
→ { "members": [{ "userId", "displayName", "role", "joinedAt", "permissions" }] }
```

```
POST   /api/v2/auth/members                              [admin]
← { "profileUrl": "https://biubiu.tools/profile/{userId}", "role": "member" }
→ { "userId", "role", "permissions" }
```

```
PATCH  /api/v2/auth/members/{userId}                     [admin]
← { "role"?, "permissions"? }
→ { "ok": true }
```

```
DELETE /api/v2/auth/members/{userId}                     [admin]
→ { "ok": true }
```

#### 1.3 Sub-Account (BIP-327)

```
POST /api/v2/auth/sub-account/derive                     [session]
← { "nonce": 0 }
→ { "subAccountAddress", "deployed": false, "eoaSigner": null }
```

```
POST /api/v2/auth/sub-account/activate                   [signed]
← { "nonce": 0, "signature": "<passkey sig>" }
→ { "subAccountAddress", "eoaSigner", "deployed", "addOwnerTx" }
```

```
GET  /api/v2/auth/sub-account/status                     [session]
?nonce=0
→ { "subAccountAddress", "deployed", "eoaSigner", "eoaSignerActive", "balance", "tokenBalances" }
```

```
POST /api/v2/auth/sub-account/revoke-signer              [signed]
← { "nonce": 0 }
→ { "ok": true }
```

### 2. Strategy (`/api/v2/strategy/*`)

Strategy CRUD, instance lifecycle, and model management. Strategies follow BIP-323 format.

#### 2.1 Strategy CRUD

```
GET    /api/v2/strategies                                [session]
```
List user's strategies in this space.
```json
→ { "strategies": [{
    "id", "name", "riskLevel", "signal": { "method" },
    "status": "draft|published",
    "instanceCount": 2,
    "createdAt"
  }] }
```

```
POST   /api/v2/strategies                                [session]
```
Create a strategy. Body is a BIP-323 JSON document.
```json
← { "$schema": "BIP-323", "id": "my-rsi", "name": "...", "signal": {...}, "entry": {...} }
→ { "id", "name", "createdAt" }
```
Engine MUST validate against BIP-323 Section 8 rules.

```
GET    /api/v2/strategies/{id}                           [session]
→ { /* Full BIP-323 JSON */ }
```

```
PUT    /api/v2/strategies/{id}                           [session]
← { /* Updated BIP-323 JSON */ }
→ { "ok": true, "version": 2 }
```
Increments strategy `version`. Running instances are NOT auto-updated.

```
DELETE /api/v2/strategies/{id}                           [session]
→ { "ok": true }
```
Fails if any instance is running. Stopped instances are deleted with the strategy.

#### 2.2 Templates & Indicators

```
GET /api/v2/strategies/templates                         [public]
```
List available strategy templates (presets for the strategy editor).
```json
→ { "templates": [{
    "id": "conservative",
    "name": "Conservative",
    "description": "Low risk, small bets, tight limits",
    "riskLevel": "conservative",
    "config": { /* BIP-323 JSON */ }
  }] }
```

```
GET /api/v2/strategies/indicators                        [public]
```
List all indicator types supported by this endpoint, with metadata for the editor UI.
```json
→ { "indicators": [{
    "type": "rsi",
    "name": "RSI",
    "category": "oscillator",
    "description": "Relative Strength Index. Overbought >70, oversold <30.",
    "config": { "period": { "type": "integer", "default": 14, "min": 5, "max": 50 } },
    "output": { "min": 0, "max": 100 },
    "defaultRules": [
      { "when": [null, 30], "up": 3, "down": -1, "label": "Oversold" },
      { "when": [70, null], "up": -1, "down": 3, "label": "Overbought" }
    ],
    "requiresData": ["klines"]
  }] }
```
This allows the editor UI to dynamically render indicator configuration without hardcoding. Third-party endpoints can add custom indicators and the UI adapts automatically.

#### 2.3 Validation

```
POST /api/v2/strategies/validate                         [public]
← { /* BIP-323 JSON */ }
→ { "valid": true, "warnings": [] }
   or
→ { "valid": false, "errors": [{ "path": "signal.params.oversold", "message": "must be < overbought" }] }
```

#### 2.4 Instance Lifecycle

```
GET    /api/v2/instances                                 [session]
```
List user's instances in this space.
```json
→ { "instances": [{
    "id", "strategyId", "label", "mode": "sandbox|live",
    "status": "stopped|starting|running|paused|error",
    "stats": { "rounds", "winRate", "profit" },
    "createdAt", "updatedAt"
  }] }
```

```
POST   /api/v2/instances                                 [signed]
← { "strategyId": "my-rsi", "label"?: "RSI Live", "mode": "sandbox" }
→ { "id": "inst_abc", "strategyId", "label", "mode", "status": "stopped" }
```
Engine checks instance limits (free: 3, member: 100, admin-configured).

```
POST   /api/v2/instances/{id}/start                      [signed]
→ { "ok": true, "status": "starting" }
```

```
POST   /api/v2/instances/{id}/stop                       [signed]
→ { "ok": true, "status": "stopped" }
```

```
POST   /api/v2/instances/{id}/pause                      [signed]
→ { "ok": true, "status": "paused" }
```

```
DELETE /api/v2/instances/{id}                            [signed]
→ { "ok": true }
```
Stops instance if running, deletes config. Historical data retained for 30 days.

```
PATCH  /api/v2/instances/{id}/config                     [signed]
← { /* Partial BIP-323 update, e.g. {"entry": {"amount": 10}} */ }
→ { "ok": true }
```
Merges with existing config. Validated against BIP-323. Instance must be stopped.

#### 2.5 Model Management

```
GET    /api/v2/models                                    [session]
```
List available models (built-in + user-uploaded).
```json
→ { "models": [{
    "id": "rf-v5-30s",
    "type": "random_forest",
    "source": "builtin|user|marketplace",
    "name", "description",
    "accuracy"?: "56%",
    "features": ["OHLCV", "RSI", "EMA"],
    "waitSec": 30
  }] }
```

```
POST   /api/v2/models/upload                             [signed]
```
Upload a custom model file (multipart/form-data).
```json
← FormData: { file: <binary>, name: "my-rf", type: "random_forest", features: ["OHLCV","RSI"] }
→ { "id": "user:user_abc:my-rf", "name", "type", "uploadedAt" }
```
Uploaded models are persisted in the space. Available for future strategy creation.

```
DELETE /api/v2/models/{id}                               [signed]
→ { "ok": true }
```
Only user-uploaded models. Fails if any running instance uses this model.


#### 3.5 Real-Time Stream

```
GET /api/v2/data/sse                                     [session]
?instance={id}&token={session_token}
```
Server-Sent Events stream per BIP-324 Event types:

| Event | Data |
|-------|------|
| `round_start` | `{ roundId, remainingSec }` |
| `signal` | `{ roundId, direction, confidence, method, score?, skipReason? }` |
| `entry` | `{ roundId, direction, price, shares, cost }` |
| `round_skip` | `{ roundId, reason }` |
| `price_update` | `{ roundId, upMid, downMid, unrealizedPnl? }` |
| `exit_trigger` | `{ roundId, exitType, price, profit }` |
| `settlement` | `{ roundId, outcome, profit }` |
| `hedge_entry` | `{ roundId, direction, price, shares }` |
| `hedge_exit` | `{ roundId, revenue }` |
| `status_change` | `{ instanceId, from, to }` |
| `heartbeat` | `{ timestamp }` |

Heartbeat: 30s. Reconnect: `retry: 3000`.

#### 3.6 Market Data

### 4. Marketplace (`/api/v2/marketplace/*`)

Strategy discovery, trading, and revenue sharing per BIP-323 economic model.

#### 4.1 Browse

```

?sort={profit24h|winRate|rounds|newest}&visibility={public|private}&page=&pageSize=&tags=&search=

    "price": 0|20,
    "riskLevel", "tags",
    "author": { "id", "name" },
    "performance": { "totalRounds", "winRate", "profit24h", "profit30d", "profitAll", "maxDrawdown" },
    "config"?: { /* BIP-323, null if private & not purchased */ },
    "followCount", "copyCount",
    "createdAt"
  }], "total", "page" }
```

```
GET /api/v2/marketplace/strategies/{id}                  [public]
→ { /* Single strategy detail, same shape as above */ }
```

#### 4.2 Publish

```
POST /api/v2/marketplace/publish                         [signed]
← { "strategyId": "my-rsi", "visibility": "public", "description"?: "..." }
→ { "ok": true, "marketplaceId": "mkt_xyz" }
```
Publishes a user's strategy to this space's marketplace.

```
PATCH /api/v2/marketplace/strategies/{id}                [signed]
← { "visibility"?: "private", "description"?: "..." }
→ { "ok": true }
```

```
DELETE /api/v2/marketplace/strategies/{id}               [signed]
→ { "ok": true }
```

#### 4.3 Follow & Copy

```
POST   /api/v2/marketplace/strategies/{id}/follow        [session]
```


```
POST /api/v2/marketplace/strategies/{id}/copy            [session]

```

| v1 Endpoint | v2 Equivalent |
|-------------|---------------|
| `GET /api/rounds` | `GET /api/v2/data/rounds?instance=default` |
| `GET /api/rounds/current` | `GET /api/v2/data/rounds/current?instance=default` |
| `GET /api/stats` | `GET /api/v2/data/stats?instance=default` |
| `GET /api/stats/daily` | `GET /api/v2/data/daily?instance=default` |
| `GET /api/stats/hourly` | `GET /api/v2/data/hourly?instance=default` |
| `GET /api/market/current` | `GET /api/v2/data/market` |
| `GET /api/strategy` | `GET /api/v2/strategies/{default}` |
| `GET /api/strategies` | `GET /api/v2/strategies` |

### 7. Error Codes

All errors follow the standard format:

```json
{ "error": { "code": "ERROR_CODE", "message": "Human-readable message" } }
```

| HTTP | Code | Description |
|------|------|-------------|
| 400 | `INVALID_PARAMS` | Request parameters invalid |
| 400 | `INVALID_STRATEGY` | BIP-323 validation failed |
| 400 | `UNSUPPORTED_VERSION` | Strategy schema version not supported |
| 401 | `UNAUTHORIZED` | Missing or invalid session |
| 401 | `SIGNATURE_INVALID` | Passkey signature verification failed |
| 403 | `FORBIDDEN` | Insufficient role/permissions |
| 403 | `INSTANCE_LIMIT` | User's instance limit reached |
| 403 | `LIVE_NOT_ALLOWED` | Live trading not enabled on this space |
| 404 | `NOT_FOUND` | Resource not found |
| 402 | `PAYMENT_REQUIRED` | Private strategy, purchase needed |
| 409 | `ALREADY_RUNNING` | Instance already in running state |
| 409 | `ALREADY_PURCHASED` | User already owns this strategy |
| 429 | `RATE_LIMITED` | Too many requests |
| 500 | `INTERNAL_ERROR` | Server error |
| 503 | `DATA_UNAVAILABLE` | Required data source unavailable |

### 8. Rate Limits

| Category | Limit | Scope |
|----------|-------|-------|
| Read endpoints | 60 req/min | Per session |
| Write endpoints | 20 req/min | Per session |
| Public endpoints | 30 req/min | Per IP |
| SSE connections | 5 concurrent | Per session |
| Model upload | 10 req/hour | Per session |

Endpoints MUST return `429` with `Retry-After` header when limits are exceeded.

## Rationale

**Why v2 prefix instead of versioning per endpoint?** A global version prefix is simpler for client SDK generation and documentation. Individual endpoint versioning leads to combinatorial complexity.

**Why separate Data and Strategy domains?** Strategy is write-heavy (CRUD, lifecycle). Data is read-heavy (charts, stats). Different caching, auth, and rate limiting strategies apply. Separating them allows independent optimization.

**Why include macro-calendar in Operations?** The calendar is infrastructure data maintained by the endpoint operator, not user-facing strategy configuration. It lives alongside health checks and resource limits.

**Why backwards compatibility with v1?** Existing frontends depend on `/api/*` routes. Breaking them would prevent gradual migration. The 6-month deprecation window gives frontend teams time to migrate.

## Security Considerations

- Write operations (`POST`, `PUT`, `DELETE`, `PATCH`) on strategy and instance endpoints MUST require passkey signature (`signed` auth level) for live mode operations. Sandbox operations MAY use session-only auth.
- Model upload MUST validate file size (<10MB) and scan for malicious content.
- SSE tokens in query parameters MUST be short-lived (< 1 hour) and single-use.
- Backup endpoints MUST be admin-only. Database files contain all trading history.
