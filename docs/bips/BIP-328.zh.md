作者: shelchin

## 摘要

定义 BiuBiu 兼容端点必须实现的 HTTP API。按五大域组织：身份、策略、数据、市场、运维。在引入 `/api/v2/*` 新能力的同时保持与现有 `/api/*` 端点的向后兼容。

## 动机

第三方端点需要清晰完整的 API 契约来接入 BiuBiu 生态。本 BIP 将现有的临时 API（btc-updown-5m-bot API.md）与 BIP-323（策略格式）、BIP-324（运行时接口）、BIP-326（身份）、BIP-327（小号）的新需求，以及通过原型确认的 Web UI 需求统一起来。

## 规范

### 0. 约定

**基础 URL：** `https://{endpoint-domain}/api/v2/`

**认证级别：**

| 级别 | 请求头 | 说明 |
|------|--------|------|
| `public` | 无 | 无需认证 |
| `session` | `Authorization: Bearer {token}` | 已登录用户（BIP-326 认证流程）|
| `signed` | `Authorization: Bearer {token}` + `X-Passkey-Signature: {sig}` | Passkey 签名的写操作 |
| `admin` | `Authorization: Bearer {token}` + 管理员角色 | 空间管理员 |

**响应格式：**

```json
{ "data": { ... } }                   // 成功
{ "error": { "code": "...", "message": "..." } }  // 错误
```

**分页：** `?page=1&pageSize=20` → 响应包含 `{ total, page, pageSize }`

**过滤：** ISO 8601 日期 `?from=&to=`，时区 `?tz_offset=`（分钟）

**CORS：** `Access-Control-Allow-Origin: *` 或配置的白名单。

### 1. 身份（`/api/v2/auth/*`）

实现 BIP-326 跨端点认证和 BIP-327 小号管理。

#### 1.1 认证

```
GET  /api/v2/auth/challenge                              [public]
```
返回一次性挑战用于 passkey 签名。5 分钟过期。
```json
→ { "challenge": "<hex>", "expiresAt": "<ISO>" }
```

```
POST /api/v2/auth/login                                  [public]
```
验证 passkey 签名，创建会话。
```json
← { "userId", "challenge", "authenticatorData", "clientDataJSON", "signature" }
→ { "token", "expiresAt", "userId", "safeAddress", "role", "permissions" }
```
引擎从 `biubiu.tools/api/identity/{userId}/public-key` 获取公钥验证。

```
POST /api/v2/auth/logout                                 [session]
→ { "ok": true }
```

```
GET  /api/v2/auth/me                                     [session]
→ { "userId", "safeAddress", "role", "permissions", "membership" }
```

#### 1.2 成员管理（管理员）

```
GET    /api/v2/auth/members                              [admin]
→ { "members": [{ "userId", "displayName", "role", "joinedAt", "permissions" }] }
```

```
POST   /api/v2/auth/members                              [admin]
← { "profileUrl": "https://biubiu.tools/profile/{userId}", "role": "member" }
→ { "userId", "role", "permissions" }
```

```
PATCH  /api/v2/auth/members/{userId}                     [admin]
← { "role"?, "permissions"? }
→ { "ok": true }
```

```
DELETE /api/v2/auth/members/{userId}                     [admin]
→ { "ok": true }
```

#### 1.3 小号（BIP-327）

```
POST /api/v2/auth/sub-account/derive                     [session]
← { "nonce": 0 }
→ { "subAccountAddress", "deployed": false, "eoaSigner": null }
```

```
POST /api/v2/auth/sub-account/activate                   [signed]
← { "nonce": 0, "signature": "<passkey 签名>" }
→ { "subAccountAddress", "eoaSigner", "deployed", "addOwnerTx" }
```

```
GET  /api/v2/auth/sub-account/status                     [session]
?nonce=0
→ { "subAccountAddress", "deployed", "eoaSigner", "eoaSignerActive", "balance", "tokenBalances" }
```

```
POST /api/v2/auth/sub-account/revoke-signer              [signed]
← { "nonce": 0 }
→ { "ok": true }
```

### 2. 策略（`/api/v2/strategy/*`）

策略 CRUD、实例生命周期、模型管理。策略遵循 BIP-323 格式。

#### 2.1 策略 CRUD

```
GET    /api/v2/strategies                                [session]
```
列出用户在此空间的策略。
```json
→ { "strategies": [{
    "id", "name", "riskLevel", "signal": { "method" },
    "status": "draft|published",
    "instanceCount": 2,
    "createdAt"
  }] }
```

```
POST   /api/v2/strategies                                [session]
```
创建策略。请求体为 BIP-323 JSON 文档。
```json
← { "$schema": "BIP-323", "id": "my-rsi", "name": "...", "signal": {...}, "entry": {...} }
→ { "id", "name", "createdAt" }
```
引擎必须按 BIP-323 第 8 节规则验证。

```
GET    /api/v2/strategies/{id}                           [session]
→ { /* 完整 BIP-323 JSON */ }
```

```
PUT    /api/v2/strategies/{id}                           [session]
← { /* 更新的 BIP-323 JSON */ }
→ { "ok": true, "version": 2 }
```
递增策略 `version`。运行中的实例不会自动更新。

```
DELETE /api/v2/strategies/{id}                           [session]
→ { "ok": true }
```
有运行中的实例时失败。已停止的实例随策略一起删除。

#### 2.2 模板与指标

```
GET /api/v2/strategies/templates                         [public]
```
列出可用策略模板（策略编辑器的预设）。
```json
→ { "templates": [{
    "id": "conservative",
    "name": "Conservative",
    "description": "低风险，小额投注，严格限制",
    "riskLevel": "conservative",
    "config": { /* BIP-323 JSON */ }
  }] }
```

```
GET /api/v2/strategies/indicators                        [public]
```
列出此端点支持的所有指标类型，附带编辑器 UI 所需的元数据。
```json
→ { "indicators": [{
    "type": "rsi",
    "name": "RSI",
    "category": "oscillator",
    "description": "相对强弱指标。超买 >70，超卖 <30。",
    "config": { "period": { "type": "integer", "default": 14, "min": 5, "max": 50 } },
    "output": { "min": 0, "max": 100 },
    "defaultRules": [
      { "when": [null, 30], "up": 3, "down": -1, "label": "Oversold" },
      { "when": [70, null], "up": -1, "down": 3, "label": "Overbought" }
    ],
    "requiresData": ["klines"]
  }] }
```
允许编辑器 UI 动态渲染指标配置，无需硬编码。第三方端点可添加自定义指标，UI 自动适配。

#### 2.3 验证

```
POST /api/v2/strategies/validate                         [public]
← { /* BIP-323 JSON */ }
→ { "valid": true, "warnings": [] }
   或
→ { "valid": false, "errors": [{ "path": "signal.params.oversold", "message": "must be < overbought" }] }
```

#### 2.4 实例生命周期

```
GET    /api/v2/instances                                 [session]
```
列出用户在此空间的实例。
```json
→ { "instances": [{
    "id", "strategyId", "label", "mode": "sandbox|live",
    "status": "stopped|starting|running|paused|error",
    "stats": { "rounds", "winRate", "profit" },
    "createdAt", "updatedAt"
  }] }
```

```
POST   /api/v2/instances                                 [signed]
← { "strategyId": "my-rsi", "label"?: "RSI Live", "mode": "sandbox" }
→ { "id": "inst_abc", "strategyId", "label", "mode", "status": "stopped" }
```
引擎检查实例限额（免费：3，会员：100，管理员配置）。

```
POST   /api/v2/instances/{id}/start                      [signed]
→ { "ok": true, "status": "starting" }
```

```
POST   /api/v2/instances/{id}/stop                       [signed]
→ { "ok": true, "status": "stopped" }
```

```
POST   /api/v2/instances/{id}/pause                      [signed]
→ { "ok": true, "status": "paused" }
```

```
DELETE /api/v2/instances/{id}                            [signed]
→ { "ok": true }
```
运行中会先停止，删除配置。历史数据保留 30 天。

```
PATCH  /api/v2/instances/{id}/config                     [signed]
← { /* BIP-323 局部更新，如 {"entry": {"amount": 10}} */ }
→ { "ok": true }
```
与现有配置合并。按 BIP-323 验证。实例必须已停止。

#### 2.5 模型管理

```
GET    /api/v2/models                                    [session]
```
列出可用模型（内置 + 用户上传）。
```json
→ { "models": [{
    "id": "rf-v5-30s",
    "type": "random_forest",
    "source": "builtin|user|marketplace",
    "name", "description",
    "accuracy"?: "56%",
    "features": ["OHLCV", "RSI", "EMA"],
    "waitSec": 30
  }] }
```

```
POST   /api/v2/models/upload                             [signed]
```
上传自定义模型文件（multipart/form-data）。
```json
← FormData: { file: <二进制>, name: "my-rf", type: "random_forest", features: ["OHLCV","RSI"] }
→ { "id": "user:user_abc:my-rf", "name", "type", "uploadedAt" }
```
上传的模型持久化在空间中，后续创建策略时可用。

```
DELETE /api/v2/models/{id}                               [signed]
→ { "ok": true }
```
仅用户上传的模型。有运行中的实例使用此模型时失败。

### 3. 数据（`/api/v2/data/*`）

只读数据，用于面板、图表和分析。映射到 BIP-324 RoundContext 输出。

#### 3.1 统计

```
GET /api/v2/data/stats                                   [session]
?instance={id}&from=&to=&hour=&tz_offset=
→ {
    "totalRounds", "entered", "skipped",
    "wins", "losses", "winRate",
    "invested", "payout", "fees", "profit",
    "avgProfit", "bestRound", "worstRound",
    "currentStreak", "maxWinStreak", "maxLoseStreak",
    "signalBet", "signalReverse", "signalSkip",
    "hedgeCost", "hedgeRevenue",
    "stopLossCount", "stopLossProfit"
  }
```

#### 3.2 轮次

```
GET /api/v2/data/rounds                                  [session]
?instance={id}&page=&pageSize=&status=&result=&signal_action=&from=&to=&hour=&tz_offset=
→ { "rows": [Round], "total", "page", "pageSize" }
```

```
GET /api/v2/data/rounds/current                          [session]
?instance={id}
→ { "round"?, "fills"?, "snapshots"?, "hedge"? }
```

```
GET /api/v2/data/rounds/{roundId}                        [session]
→ { "round", "fills", "snapshots", "hedge", "diagnostics"? }
```
`diagnostics` 包含 BIP-324 SignalOutput.diagnostics（如可用）。

#### 3.3 时间序列

```
GET /api/v2/data/hourly                                  [session]
?instance={id}&from=&to=
→ [{ "hour", "rounds", "wins", "losses", "winRate", "invested", "payout", "profit", "avgProfit" }]
```

```
GET /api/v2/data/daily                                   [session]
?instance={id}&from=&to=&tz_offset=
→ [{ "date", "rounds", "entered", "skipped", "wins", "losses", "winRate", "invested", "payout", "profit" }]
```

```
GET /api/v2/data/weekday                                 [session]
?instance={id}&from=&to=&tz_offset=
→ [{ "day": 0, "label": "Mon", "rounds", "wins", "losses", "winRate", "profit", "avgProfit" }]
```

#### 3.4 收益摘要（侧栏对比）

```
GET /api/v2/data/profits                                 [session]
?instances={id1,id2,...}
→ {
    "{id1}": {
      "current": { "status": "entered|settled|watching" },
      "hour":  { "profit", "rounds", "winRate" },
      "day":   { "profit", "rounds", "winRate" },
      "all":   { "profit", "rounds", "winRate" }
    },
    ...
  }
```

#### 3.5 实时流

```
GET /api/v2/data/sse                                     [session]
?instance={id}&token={session_token}
```
遵循 BIP-324 事件类型的 Server-Sent Events 流：

| 事件 | 数据 |
|------|------|
| `round_start` | `{ roundId, remainingSec }` |
| `signal` | `{ roundId, direction, confidence, method, score?, skipReason? }` |
| `entry` | `{ roundId, direction, price, shares, cost }` |
| `round_skip` | `{ roundId, reason }` |
| `price_update` | `{ roundId, upMid, downMid, unrealizedPnl? }` |
| `exit_trigger` | `{ roundId, exitType, price, profit }` |
| `settlement` | `{ roundId, outcome, profit }` |
| `hedge_entry` | `{ roundId, direction, price, shares }` |
| `hedge_exit` | `{ roundId, revenue }` |
| `status_change` | `{ instanceId, from, to }` |
| `heartbeat` | `{ timestamp }` |

心跳：30 秒。重连：`retry: 3000`。

#### 3.6 市场数据

```
GET /api/v2/data/market                                  [public]
→ {
    "roundId", "slug", "remainingSec",
    "up": { "bestBid", "bestAsk", "mid" },
    "down": { "bestBid", "bestAsk", "mid" }
  }
```

### 4. 市场（`/api/v2/marketplace/*`）

策略发现、交易和收益分成。

#### 4.1 浏览

```
GET /api/v2/marketplace/strategies                       [public]
?sort={profit24h|winRate|rounds|newest}&visibility={public|private}&page=&pageSize=&tags=&search=
→ { "strategies": [{
    "id", "name", "description", "visibility",
    "price": 0|20,
    "riskLevel", "tags",
    "author": { "id", "name" },
    "performance": { "totalRounds", "winRate", "profit24h", "profit30d", "profitAll", "maxDrawdown" },
    "config"?: { /* BIP-323，私有且未购买时为 null */ },
    "followCount", "copyCount",
    "createdAt"
  }], "total", "page" }
```

```
GET /api/v2/marketplace/strategies/{id}                  [public]
→ { /* 单个策略详情，同上 */ }
```

#### 4.2 发布

```
POST /api/v2/marketplace/publish                         [signed]
← { "strategyId": "my-rsi", "visibility": "public", "description"?: "..." }
→ { "ok": true, "marketplaceId": "mkt_xyz" }
```
将用户策略发布到此空间的市场。

```
PATCH /api/v2/marketplace/strategies/{id}                [signed]
← { "visibility"?: "private", "description"?: "..." }
→ { "ok": true }
```

```
DELETE /api/v2/marketplace/strategies/{id}               [signed]
→ { "ok": true }
```
取消发布。已有的复制/关注不受影响。

#### 4.3 关注与复制

```
POST   /api/v2/marketplace/strategies/{id}/follow        [session]
→ { "ok": true }
```
订阅表现更新。私有策略无法获取配置。

```
DELETE /api/v2/marketplace/strategies/{id}/follow        [session]
→ { "ok": true }
```

```
POST /api/v2/marketplace/strategies/{id}/copy            [session]
→ { "strategyId": "copy-of-my-rsi", "ok": true }
```
将配置复制到用户的策略中。私有策略需要先购买。

#### 4.4 导出与导入

```
GET  /api/v2/marketplace/strategies/{id}/export          [session]
→ BIP-323 JSON 文件（Content-Disposition: attachment）
```
私有策略：仅已购买或拥有者可导出。

```
POST /api/v2/marketplace/import                          [session]
← { "json": { /* BIP-323 */ } }
   或
← { "shareLink": "https://biubiu.tools/s/{shortId}" }
→ { "strategyId": "imported-xyz", "ok": true }
```

#### 4.5 购买与验证

```
POST /api/v2/marketplace/purchase/verify                 [session]
← { "strategyId": "mkt_xyz", "txHash": "0x...", "buyer": "0x..." }
→ { "verified": true, "config": { /* 解锁的 BIP-323 JSON */ } }
```
验证链上购买并解锁策略配置。

```
GET /api/v2/marketplace/purchases                        [session]
→ { "purchases": [{ "strategyId", "txHash", "price", "purchasedAt" }] }
```

#### 4.6 排行榜

```
GET /api/v2/marketplace/leaderboard                      [public]
?period={24h|7d|30d}&limit=20
→ { "strategies": [{
    "rank", "id", "name", "author",
    "winRate", "profit", "rounds"
  }] }
```

| v1 端点 | v2 等价 |
|---------|---------|

| `GET /api/rounds/current` | `GET /api/v2/data/rounds/current?instance=default` |
| `GET /api/stats` | `GET /api/v2/data/stats?instance=default` |
| `GET /api/stats/daily` | `GET /api/v2/data/daily?instance=default` |
| `GET /api/stats/hourly` | `GET /api/v2/data/hourly?instance=default` |
| `GET /api/market/current` | `GET /api/v2/data/market` |
| `GET /api/strategy` | `GET /api/v2/strategies/{default}` |
| `GET /api/strategies` | `GET /api/v2/strategies` |
| `GET /api/health` | `GET /api/v2/ops/health` |

### 7. 错误码

所有错误遵循标准格式：

```json
{ "error": { "code": "ERROR_CODE", "message": "人类可读的消息" } }
```

| HTTP | 代码 | 说明 |
|------|------|------|
| 400 | `INVALID_PARAMS` | 请求参数无效 |
| 400 | `INVALID_STRATEGY` | BIP-323 验证失败 |
| 400 | `UNSUPPORTED_VERSION` | 策略 schema 版本不支持 |
| 401 | `UNAUTHORIZED` | 缺少或无效的会话 |
| 401 | `SIGNATURE_INVALID` | Passkey 签名验证失败 |
| 403 | `FORBIDDEN` | 角色/权限不足 |
| 403 | `INSTANCE_LIMIT` | 用户实例限额已达 |
| 403 | `LIVE_NOT_ALLOWED` | 此空间未启用实盘交易 |
| 404 | `NOT_FOUND` | 资源不存在 |
| 402 | `PAYMENT_REQUIRED` | 私有策略，需要购买 |
| 409 | `ALREADY_RUNNING` | 实例已在运行状态 |
| 409 | `ALREADY_PURCHASED` | 用户已拥有此策略 |
| 429 | `RATE_LIMITED` | 请求过于频繁 |
| 500 | `INTERNAL_ERROR` | 服务器错误 |
| 503 | `DATA_UNAVAILABLE` | 所需数据源不可用 |

### 8. 频率限制

| 类别 | 限制 | 范围 |
|------|------|------|
| 读取端点 | 60 次/分 | 每会话 |
| 写入端点 | 20 次/分 | 每会话 |
| 公开端点 | 30 次/分 | 每 IP |
| SSE 连接 | 5 个并发 | 每会话 |
| 模型上传 | 10 次/时 | 每会话 |

超过限制时端点必须返回 `429` 并附带 `Retry-After` 头。

## 设计理由

**为什么用 v2 前缀而不是逐端点版本化？** 全局版本前缀对客户端 SDK 生成和文档更简单。逐端点版本化导致组合复杂度爆炸。

**为什么分离数据域和策略域？** 策略是写密集的（CRUD、生命周期）。数据是读密集的（图表、统计）。不同的缓存、认证和频率限制策略适用。分离允许独立优化。

**为什么宏观日历在运维域？** 日历是端点运营者维护的基础设施数据，不是用户面向的策略配置。它与健康检查和资源限制放在一起。

**为什么保持与 v1 的向后兼容？** 现有前端依赖 `/api/*` 路由。破坏它们会阻碍渐进式迁移。6 个月的弃用窗口给前端团队迁移时间。

## 安全考虑

- 策略和实例端点的写操作（`POST`、`PUT`、`DELETE`、`PATCH`）在实盘模式下必须要求 passkey 签名（`signed` 认证级别）。沙盘操作可以仅用会话认证。
- 模型上传必须验证文件大小（<10MB）并扫描恶意内容。
- 查询参数中的 SSE token 必须短期有效（< 1 小时）且一次性使用。
- 备份端点必须仅限管理员。数据库文件包含所有交易历史。
