## Abstract

Defines the key management architecture for EOA signer private keys used in BIP-327 sub-account automation. Establishes the invariant: **plaintext keys exist only in process memory, never on disk or in transit**. Specifies the master key lifecycle, encryption scheme, backup to Cloudflare R2, and recovery procedures.

## Motivation

BIP-327 delegates signing authority to an EOA keypair held by the endpoint. This private key can move funds in a user's sub-account. A leak means direct financial loss. BIP-327 Section 7 states "private keys encrypted with AES-256-GCM" but does not specify: where the encryption key comes from, how it is stored, how ciphertexts are backed up, or how recovery works after data loss. This BIP fills that gap.

## Specification

### 1. Security Invariants

These invariants MUST hold at all times:

| Invariant | Description |
|-----------|-------------|
| **I1** | EOA private keys MUST NOT exist in plaintext on any persistent storage (disk, database, cloud storage, logs) |
| **I2** | EOA private keys MAY exist in plaintext only in process memory, and only for the duration of a signing operation |
| **I3** | The master key MUST NOT be persisted on disk on the endpoint server |
| **I4** | Ciphertext backups are safe to store on untrusted storage (R2, S3) — they are useless without the master key |
| **I5** | Loss of the master key means loss of all EOA keys — this is by design (fail-secure) |

### 3. Master Key

The master key is a 256-bit symmetric key used to encrypt/decrypt all EOA private keys on this endpoint.

#### 3.1 Generation

```
masterKey = crypto.getRandomValues(new Uint8Array(32))
```

Generated once when the endpoint is first set up. The operator MUST store it securely outside the endpoint server.

#### 3.2 Sources

On every process startup, the endpoint fetches the master key from one of these sources:

| Source | Config | Security Level | Use Case |
|--------|--------|---------------|----------|
| Environment variable | `MASTER_KEY=hex` | Low | Development, single-server |
| Interactive prompt | stdin on startup | Medium | Small operators, manual restart |
| Cloud KMS | `KMS_KEY_ARN=arn:...` | High | AWS/GCP hosted endpoints |
| HashiCorp Vault | `VAULT_ADDR` + `VAULT_TOKEN` | High | Self-hosted infrastructure |

**Cloud KMS mode (recommended for production):**

In this mode, the master key never leaves the KMS. Instead of fetching the master key, the endpoint sends encrypt/decrypt requests to the KMS API:

```
// Encrypt
ciphertext = await kms.encrypt({
  KeyId: keyArn,
  Plaintext: eoaPrivateKey
})

// Decrypt
plaintext = await kms.decrypt({
  KeyId: keyArn,
  CiphertextBlob: ciphertext
})
```

The master key exists only inside the KMS hardware. Even the endpoint operator cannot extract it.

**Environment variable mode (simplest):**

```
// .env (NOT committed to git)
MASTER_KEY=0a1b2c3d...64hex...

// Startup
const masterKey = Buffer.from(process.env.MASTER_KEY, 'hex')
// masterKey is now in memory only
```

#### 3.3 Memory Lifecycle

```
Process start:
  → Load master key from source into memory
  → masterKey is a Buffer/Uint8Array in heap

Process running:
  → masterKey stays in memory
  → Used for encrypt/decrypt operations

Process shutdown:
  → masterKey.fill(0)  // Overwrite with zeros
  → masterKey = null
  → Force garbage collection if possible
```

### 4. EOA Key Encryption

#### 4.1 Scheme

Algorithm: **AES-256-GCM** (authenticated encryption)

```typescript
interface EncryptedKey {
  version: 1;                    // Schema version for future migration
  algorithm: "aes-256-gcm";
  iv: string;                   // 12 bytes, hex-encoded
  ciphertext: string;           // Encrypted private key, hex-encoded
  tag: string;                  // 16-byte GCM auth tag, hex-encoded
  subAccountId: string;         // Which sub-account this key belongs to
  eoaAddress: string;           // Public address (for verification after decrypt)
  createdAt: string;            // ISO 8601
}
```

#### 4.2 Encrypt (on key creation)

```typescript
function encryptEoaKey(
  privateKey: Uint8Array,        // 32 bytes, plaintext
  masterKey: Uint8Array,         // 32 bytes, from memory
  subAccountId: string,
  eoaAddress: string
): EncryptedKey {
  const iv = crypto.getRandomValues(new Uint8Array(12));

  const encrypted = crypto.subtle.encrypt(
    { name: "AES-GCM", iv, tagLength: 128 },
    masterKey,
    privateKey
  );

  // Immediately zero out the plaintext key
  privateKey.fill(0);

  return {
    version: 1,
    algorithm: "aes-256-gcm",
    iv: hex(iv),
    ciphertext: hex(encrypted.slice(0, -16)),
    tag: hex(encrypted.slice(-16)),
    subAccountId,
    eoaAddress,
    createdAt: new Date().toISOString()
  };
}
```

#### 4.3 Decrypt (on signing)

```typescript
function decryptEoaKey(
  encrypted: EncryptedKey,
  masterKey: Uint8Array
): Uint8Array {
  const iv = fromHex(encrypted.iv);
  const data = concat(fromHex(encrypted.ciphertext), fromHex(encrypted.tag));

  const plaintext = crypto.subtle.decrypt(
    { name: "AES-GCM", iv, tagLength: 128 },
    masterKey,
    data
  );

  // Verify: derive address from decrypted key, compare with stored eoaAddress
  const derivedAddress = privateKeyToAddress(plaintext);
  if (derivedAddress !== encrypted.eoaAddress) {
    throw new Error("Key integrity check failed: address mismatch");
  }

  return plaintext;  // Caller MUST zero this after use
}
```

### 5. Storage

#### 5.1 Local Disk

Encrypted keys stored as individual JSON files:

```
/data/keys/
├── 0x2222_nonce0.enc.json      ← Sub-account 0x2222, nonce 0
├── 0x3333_nonce0.enc.json
└── 0x3333_nonce1.enc.json
```

Each file contains one `EncryptedKey` JSON object.

File permissions: `0600` (owner read/write only).

Directory permissions: `0700`.

#### 5.2 Cloudflare R2 Backup

On every key creation or rotation, the ciphertext is backed up to R2:

```
Bucket:  biubiu-keys-backup
Path:    /{endpointId}/{subAccountId}_nonce{N}.enc.json
```

R2 configuration:

| Setting | Value |
|---------|-------|
| Versioning | Enabled (keep last 5 versions) |
| Lifecycle | Delete versions older than 90 days |
| Access | Private (endpoint's R2 API token only) |
| Encryption | Default (R2 encrypts at rest with managed keys) |

**Backup is ciphertext only.** Even if R2 is fully compromised, attacker gets encrypted blobs that are useless without the master key.

#### 5.3 Backup Sync Flow

```
Key created:
  1. Generate keypair in memory
  2. Encrypt with master key → EncryptedKey
  3. Write to local disk: /data/keys/{id}.enc.json
  4. Upload to R2: /{endpointId}/{id}.enc.json
  5. Verify R2 upload succeeded
  6. Return EOA address to caller

If R2 upload fails:
  → Log warning, retry 3 times with exponential backoff
  → If still fails: key exists locally but no backup
  → Surface warning in /api/v2/ops/health: "backup_degraded"
  → Do NOT block key creation (availability > backup)
```

### 6. Signing Flow

```
Strategy needs to sign a transaction:

  1. Check in-memory cache for decrypted key
     → If cached and TTL valid: use cached key, skip to step 4

  2. Read EncryptedKey from /data/keys/{subAccountId}.enc.json

  3. Decrypt with master key (in memory)
     → plaintext = decryptEoaKey(encrypted, masterKey)

  4. Sign transaction
     → signature = sign(txHash, plaintext)

  5. Optionally cache plaintext key in memory (TTL: 5 minutes)
     → Reduces disk reads for frequent signing
     → Cache entry auto-cleared after TTL

  6. If not caching: zero out plaintext immediately
     → plaintext.fill(0)
```

**In-memory cache considerations:**

| Setting | Value | Rationale |
|---------|-------|-----------|
| TTL | 5 minutes | Covers one full round (5-min prediction cycle) |
| Max entries | 100 | Prevents unbounded memory growth |
| Eviction | LRU | Least recently used keys evicted first |
| Shutdown | Clear all | `cache.clear()` + overwrite buffers with zeros |

Cache is OPTIONAL. Endpoints with strict security requirements MAY disable it and decrypt from disk on every sign operation.

### 7. Key Rotation

#### 7.1 Master Key Rotation

When the master key needs to change (compromise suspected, policy requirement):

```
1. Load new master key into memory (from new source)
2. For each EncryptedKey on disk:
   a. Decrypt with OLD master key
   b. Re-encrypt with NEW master key
   c. Write new ciphertext to disk (atomic replace)
   d. Upload new ciphertext to R2
   e. Zero out plaintext
3. Update master key source (env var, KMS, etc.)
4. Replace old master key in memory with new one
5. Zero out old master key
```

All steps MUST be atomic per key. If rotation fails mid-way, some keys use old master key and some use new. The endpoint MUST track which version each key uses:

```json
{
  "version": 1,
  "masterKeyVersion": 2,        // ← Added during rotation
  "algorithm": "aes-256-gcm",
  ...
}
```

#### 7.2 EOA Key Rotation

When an EOA key needs to change (BIP-327 re-keying):

```
1. Generate new EOA keypair
2. Encrypt new key → store locally + R2
3. User signs Safe.swapOwner(oldEOA, newEOA) with passkey
4. Verify new EOA is now owner on-chain
5. Delete old key file locally + R2
6. Clear old key from memory cache
```

### 8. Recovery

#### 8.1 Local Disk Loss

If the endpoint's disk is lost but master key is available:

```
1. Start endpoint with master key
2. Download all .enc.json files from R2
3. Decrypt each to verify integrity
4. Resume normal operation
```

#### 8.2 Master Key Loss

If the master key is lost:

```
ALL EOA KEYS ARE PERMANENTLY LOST.
This is by design (fail-secure).

Recovery:
1. For each sub-account:
   a. User signs Safe.removeOwner(oldEOA) with passkey
   b. Endpoint generates new EOA keypair
   c. User signs Safe.addOwner(newEOA) with passkey
2. This is manual and requires user interaction for each sub-account
```

Endpoints MUST document their master key backup procedure. Recommended: store master key in a separate system (password manager, separate KMS, physical safe).

#### 8.3 R2 Access Loss

If R2 becomes inaccessible:

```
1. Local disk keys continue to work (primary source)
2. New keys are created but not backed up
3. /api/v2/ops/health reports "backup_degraded"
4. When R2 recovers: sync all local keys to R2
```

### 9. Audit Logging

All key operations MUST be logged:

| Event | Log Fields |
|-------|-----------|
| `key_created` | subAccountId, eoaAddress, timestamp |
| `key_decrypted` | subAccountId, eoaAddress, purpose ("sign_tx"), timestamp |
| `key_rotated` | subAccountId, oldAddress, newAddress, timestamp |
| `key_deleted` | subAccountId, eoaAddress, timestamp |
| `master_key_loaded` | source ("env", "kms", "vault", "prompt"), timestamp |
| `master_key_rotated` | keysReEncrypted, timestamp |
| `backup_uploaded` | subAccountId, r2Path, timestamp |
| `backup_restored` | subAccountId, r2Path, timestamp |

Logs MUST NOT contain: private keys, master key, ciphertext content, or any material that could aid decryption.

### 10. Implementation Checklist

Endpoint operators MUST verify all items before enabling live trading:

- [ ] Master key source configured and tested
- [ ] Master key stored in a separate backup system
- [ ] `/data/keys/` directory exists with permissions `0700`
- [ ] R2 bucket created with versioning enabled
- [ ] R2 API credentials configured (scoped to single bucket)
- [ ] Encrypt/decrypt cycle tested (create key → encrypt → decrypt → verify address)
- [ ] Backup sync tested (create key → verify appears in R2)
- [ ] Recovery tested (delete local key → restore from R2 → verify signing works)
- [ ] Audit logging enabled and tested
- [ ] `/api/v2/ops/health` reports `keyManagement: "ok"`

## Rationale

**Why AES-256-GCM over other schemes?** GCM provides authenticated encryption (integrity + confidentiality) in a single operation. The auth tag prevents tampering with ciphertexts. AES-256 is the standard for symmetric encryption of high-value secrets.

**Why individual files per key?** Atomic operations: creating, rotating, or deleting one key doesn't risk corrupting others. A single encrypted database (like SQLCipher) would couple all keys to one lock, and corruption could lose all keys at once.

**Why R2 over other backup targets?** R2 has no egress fees (backups can be large over time), S3-compatible API (portable), and built-in versioning. The ciphertext-only invariant means any cloud storage is safe — the security boundary is the master key, not the storage.

**Why allow in-memory cache?** A 5-minute prediction market requires signing every round. Decrypting from disk 12 times per hour per sub-account is unnecessary I/O. The cache trades a small memory footprint for significant latency reduction. The TTL ensures plaintext doesn't linger indefinitely.

**Why fail-secure on master key loss?** The alternative (fail-open: store a recovery backdoor) introduces a permanent vulnerability. Users can always recover via passkey + Safe.addOwner flow. Lost master key = operational disruption, not permanent fund loss.

## Security Considerations

- Master key in environment variables is the weakest option. Process dumps, `/proc/environ`, and container inspection can expose it. Cloud KMS is strongly recommended for production.
- In-memory cache creates a window where plaintext keys exist beyond the signing operation. Endpoints with strict security MAY disable caching.
- R2 API credentials MUST be scoped to the single backup bucket with write-only permissions. Read access is only needed during recovery.
- Audit logs may be subpoenaed. Ensure they contain no key material.
- The `eoaAddress` field in `EncryptedKey` is a post-decrypt integrity check. An attacker who modifies the ciphertext cannot produce valid plaintext that matches the stored address.
