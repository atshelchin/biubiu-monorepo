## 摘要

定义 BIP-327 小号自动化中 EOA 签名者私钥的管理架构。建立不变量：**明文密钥仅存在于进程内存中，永远不在磁盘或传输中出现**。规定主密钥生命周期、加密方案、Cloudflare R2 备份、以及恢复流程。

## 动机

BIP-327 将签名权委托给端点持有的 EOA 密钥对。该私钥可以转移用户小号中的资金，泄漏即直接经济损失。BIP-327 第 7 节写了"用 AES-256-GCM 加密"但未规定：加密密钥从哪来、怎么存储、密文如何备份、数据丢失后怎么恢复。本 BIP 填补该空白。

## 规范

### 1. 安全不变量

以下不变量必须在任何时候成立：

| 不变量 | 描述 |
|--------|------|
| **I1** | EOA 私钥不得以明文存在于任何持久化存储（磁盘、数据库、云存储、日志）|
| **I2** | EOA 私钥仅可以明文存在于进程内存中，且仅在签名操作期间 |
| **I3** | 主密钥不得持久化到端点服务器的磁盘上 |
| **I4** | 密文备份可以安全存储在不受信存储上（R2、S3）——没有主密钥它们毫无用处 |
| **I5** | 丢失主密钥意味着丢失所有 EOA 密钥——这是设计意图（故障安全）|

### 3. 主密钥

主密钥是 256 位对称密钥，用于加密/解密此端点上的所有 EOA 私钥。

#### 3.1 生成

```
masterKey = crypto.getRandomValues(new Uint8Array(32))
```

端点首次设置时生成一次。运营者必须在端点服务器之外安全存储。

#### 3.2 来源

每次进程启动时，端点从以下来源之一获取主密钥：

| 来源 | 配置 | 安全级别 | 适用场景 |
|------|------|----------|----------|
| 环境变量 | `MASTER_KEY=hex` | 低 | 开发、单机 |
| 交互式输入 | 启动时 stdin | 中 | 小型运营者，手动重启 |
| Cloud KMS | `KMS_KEY_ARN=arn:...` | 高 | AWS/GCP 托管端点 |
| HashiCorp Vault | `VAULT_ADDR` + `VAULT_TOKEN` | 高 | 自建基础设施 |

**Cloud KMS 模式（生产推荐）：**

此模式下，主密钥永远不离开 KMS。端点不获取主密钥本身，而是向 KMS API 发送加密/解密请求：

```
// 加密
ciphertext = await kms.encrypt({
  KeyId: keyArn,
  Plaintext: eoaPrivateKey
})

// 解密
plaintext = await kms.decrypt({
  KeyId: keyArn,
  CiphertextBlob: ciphertext
})
```

主密钥仅存在于 KMS 硬件内部。端点运营者也无法提取。

**环境变量模式（最简单）：**

```
// .env（不提交到 git）
MASTER_KEY=0a1b2c3d...64hex...

// 启动时
const masterKey = Buffer.from(process.env.MASTER_KEY, 'hex')
// masterKey 现在仅在内存中
```

#### 3.3 内存生命周期

```
进程启动：
  → 从来源加载主密钥到内存
  → masterKey 是堆上的 Buffer/Uint8Array

进程运行中：
  → masterKey 留在内存
  → 用于加密/解密操作

进程关闭：
  → masterKey.fill(0)  // 用零覆盖
  → masterKey = null
  → 可能的话强制垃圾回收
```

### 4. EOA 密钥加密

#### 4.1 方案

算法：**AES-256-GCM**（认证加密）

```typescript
interface EncryptedKey {
  version: 1;                    // Schema 版本，用于未来迁移
  algorithm: "aes-256-gcm";
  iv: string;                   // 12 字节，hex 编码
  ciphertext: string;           // 加密的私钥，hex 编码
  tag: string;                  // 16 字节 GCM 认证标签，hex 编码
  subAccountId: string;         // 此密钥属于哪个小号
  eoaAddress: string;           // 公开地址（解密后验证用）
  createdAt: string;            // ISO 8601
}
```

#### 4.2 加密（创建密钥时）

```typescript
function encryptEoaKey(
  privateKey: Uint8Array,        // 32 字节，明文
  masterKey: Uint8Array,         // 32 字节，来自内存
  subAccountId: string,
  eoaAddress: string
): EncryptedKey {
  const iv = crypto.getRandomValues(new Uint8Array(12));

  const encrypted = crypto.subtle.encrypt(
    { name: "AES-GCM", iv, tagLength: 128 },
    masterKey,
    privateKey
  );

  // 立即将明文密钥清零
  privateKey.fill(0);

  return {
    version: 1,
    algorithm: "aes-256-gcm",
    iv: hex(iv),
    ciphertext: hex(encrypted.slice(0, -16)),
    tag: hex(encrypted.slice(-16)),
    subAccountId,
    eoaAddress,
    createdAt: new Date().toISOString()
  };
}
```

#### 4.3 解密（签名时）

```typescript
function decryptEoaKey(
  encrypted: EncryptedKey,
  masterKey: Uint8Array
): Uint8Array {
  const iv = fromHex(encrypted.iv);
  const data = concat(fromHex(encrypted.ciphertext), fromHex(encrypted.tag));

  const plaintext = crypto.subtle.decrypt(
    { name: "AES-GCM", iv, tagLength: 128 },
    masterKey,
    data
  );

  // 验证：从解密的密钥派生地址，与存储的 eoaAddress 对比
  const derivedAddress = privateKeyToAddress(plaintext);
  if (derivedAddress !== encrypted.eoaAddress) {
    throw new Error("密钥完整性检查失败：地址不匹配");
  }

  return plaintext;  // 调用方必须在使用后清零
}
```

### 5. 存储

#### 5.1 本地磁盘

加密密钥以独立 JSON 文件存储：

```
/data/keys/
├── 0x2222_nonce0.enc.json      ← 小号 0x2222，nonce 0
├── 0x3333_nonce0.enc.json
└── 0x3333_nonce1.enc.json
```

每个文件包含一个 `EncryptedKey` JSON 对象。

文件权限：`0600`（仅所有者可读写）。

目录权限：`0700`。

#### 5.2 Cloudflare R2 备份

每次密钥创建或轮换时，密文备份到 R2：

```
Bucket:  biubiu-keys-backup
Path:    /{endpointId}/{subAccountId}_nonce{N}.enc.json
```

R2 配置：

| 设置 | 值 |
|------|------|
| 版本控制 | 启用（保留最近 5 个版本）|
| 生命周期 | 删除 90 天以上的版本 |
| 访问 | 私有（仅端点的 R2 API token）|
| 加密 | 默认（R2 用托管密钥静态加密）|

**备份仅有密文。** 即使 R2 被完全入侵，攻击者得到的加密数据块没有主密钥毫无用处。

#### 5.3 备份同步流程

```
密钥创建：
  1. 在内存中生成密钥对
  2. 用主密钥加密 → EncryptedKey
  3. 写入本地磁盘：/data/keys/{id}.enc.json
  4. 上传到 R2：/{endpointId}/{id}.enc.json
  5. 验证 R2 上传成功
  6. 返回 EOA 地址给调用方

R2 上传失败时：
  → 记录警告，指数退避重试 3 次
  → 仍失败：密钥在本地存在但无备份
  → 在 /api/v2/ops/health 中标记："backup_degraded"
  → 不阻断密钥创建（可用性 > 备份）
```

### 6. 签名流程

```
策略需要签署交易：

  1. 检查内存缓存中是否有解密的密钥
     → 如果有且 TTL 有效：使用缓存密钥，跳到步骤 4

  2. 从 /data/keys/{subAccountId}.enc.json 读取 EncryptedKey

  3. 用主密钥解密（在内存中）
     → plaintext = decryptEoaKey(encrypted, masterKey)

  4. 签署交易
     → signature = sign(txHash, plaintext)

  5. 可选：将明文密钥缓存到内存（TTL：5 分钟）
     → 减少频繁签名的磁盘读取
     → 缓存条目在 TTL 后自动清除

  6. 如不缓存：立即清零明文
     → plaintext.fill(0)
```

**内存缓存配置：**

| 设置 | 值 | 原因 |
|------|------|------|
| TTL | 5 分钟 | 覆盖一个完整轮次（5 分钟预测周期）|
| 最大条目 | 100 | 防止内存无限增长 |
| 淘汰策略 | LRU | 最近最少使用的密钥优先淘汰 |
| 关闭 | 全部清除 | `cache.clear()` + 用零覆盖缓冲区 |

缓存是可选的。有严格安全要求的端点可以禁用它，每次签名都从磁盘解密。

### 7. 密钥轮换

#### 7.1 主密钥轮换

当主密钥需要更换时（怀疑泄漏、策略要求）：

```
1. 将新主密钥加载到内存（从新来源）
2. 对磁盘上的每个 EncryptedKey：
   a. 用旧主密钥解密
   b. 用新主密钥重新加密
   c. 写入新密文到磁盘（原子替换）
   d. 上传新密文到 R2
   e. 清零明文
3. 更新主密钥来源（环境变量、KMS 等）
4. 用新主密钥替换内存中的旧主密钥
5. 清零旧主密钥
```

每个密钥的所有步骤必须是原子的。如果轮换中途失败，部分密钥用旧主密钥，部分用新主密钥。端点必须追踪每个密钥使用的版本：

```json
{
  "version": 1,
  "masterKeyVersion": 2,        // ← 轮换时添加
  "algorithm": "aes-256-gcm",
  ...
}
```

#### 7.2 EOA 密钥轮换

当 EOA 密钥需要更换时（BIP-327 重新生成密钥）：

```
1. 生成新 EOA 密钥对
2. 加密新密钥 → 存储到本地 + R2
3. 用户用 passkey 签署 Safe.swapOwner(oldEOA, newEOA)
4. 验证新 EOA 已成为链上所有者
5. 删除本地 + R2 的旧密钥文件
6. 清除内存缓存中的旧密钥
```

### 8. 恢复

#### 8.1 本地磁盘丢失

磁盘丢失但主密钥可用时：

```
1. 用主密钥启动端点
2. 从 R2 下载所有 .enc.json 文件
3. 解密每个文件验证完整性
4. 恢复正常运行
```

#### 8.2 主密钥丢失

主密钥丢失时：

```
所有 EOA 密钥永久丢失。
这是设计意图（故障安全）。

恢复流程：
1. 对每个小号：
   a. 用户用 passkey 签署 Safe.removeOwner(oldEOA)
   b. 端点生成新 EOA 密钥对
   c. 用户用 passkey 签署 Safe.addOwner(newEOA)
2. 这是手动的，需要每个小号都有用户交互
```

端点必须文档化其主密钥备份流程。推荐：将主密钥存储在独立系统中（密码管理器、独立 KMS、实体保险柜）。

#### 8.3 R2 访问丢失

R2 不可访问时：

```
1. 本地磁盘密钥继续工作（主要来源）
2. 新密钥被创建但未备份
3. /api/v2/ops/health 报告 "backup_degraded"
4. R2 恢复后：将所有本地密钥同步到 R2
```

### 9. 审计日志

所有密钥操作必须记录日志：

| 事件 | 日志字段 |
|------|----------|
| `key_created` | subAccountId, eoaAddress, timestamp |
| `key_decrypted` | subAccountId, eoaAddress, purpose ("sign_tx"), timestamp |
| `key_rotated` | subAccountId, oldAddress, newAddress, timestamp |
| `key_deleted` | subAccountId, eoaAddress, timestamp |
| `master_key_loaded` | source ("env", "kms", "vault", "prompt"), timestamp |
| `master_key_rotated` | keysReEncrypted, timestamp |
| `backup_uploaded` | subAccountId, r2Path, timestamp |
| `backup_restored` | subAccountId, r2Path, timestamp |

日志不得包含：私钥、主密钥、密文内容、或任何可辅助解密的材料。

### 10. 实施清单

端点运营者在启用实盘交易前必须验证所有条目：

- [ ] 主密钥来源已配置并测试
- [ ] 主密钥已存储在独立备份系统中
- [ ] `/data/keys/` 目录存在且权限为 `0700`
- [ ] R2 bucket 已创建且启用版本控制
- [ ] R2 API 凭证已配置（限定到单个 bucket）
- [ ] 加密/解密循环已测试（创建密钥 → 加密 → 解密 → 验证地址）
- [ ] 备份同步已测试（创建密钥 → 验证出现在 R2）
- [ ] 恢复已测试（删除本地密钥 → 从 R2 恢复 → 验证签名正常）
- [ ] 审计日志已启用并测试
- [ ] `/api/v2/ops/health` 报告 `keyManagement: "ok"`

## 设计理由

**为什么用 AES-256-GCM？** GCM 在单次操作中提供认证加密（完整性 + 机密性）。认证标签防止篡改密文。AES-256 是高价值密钥对称加密的标准。

**为什么每个密钥独立文件？** 原子操作：创建、轮换或删除一个密钥不会风险破坏其他密钥。单个加密数据库（如 SQLCipher）会将所有密钥耦合到一个锁上，损坏可能一次丢失所有密钥。

**为什么用 R2？** R2 无出口费用（备份随时间可能很大）、S3 兼容 API（可移植）、内置版本控制。仅密文的不变量意味着任何云存储都是安全的——安全边界是主密钥，不是存储。

**为什么允许内存缓存？** 5 分钟预测市场每轮都需要签名。每小时每个小号从磁盘解密 12 次是不必要的 I/O。缓存用少量内存换取显著的延迟降低。TTL 确保明文不会无限期滞留。

**为什么主密钥丢失时故障安全？** 替代方案（故障开放：存储恢复后门）引入永久漏洞。用户始终可以通过 passkey + Safe.addOwner 流程恢复。主密钥丢失 = 运营中断，不是永久资金丢失。

## 安全考虑

- 环境变量中的主密钥是最弱的方案。进程转储、`/proc/environ` 和容器检查可以暴露它。生产环境强烈推荐 Cloud KMS。
- 内存缓存创建了一个明文密钥在签名操作之外存在的窗口。有严格安全要求的端点可以禁用缓存。
- R2 API 凭证必须限定到单个备份 bucket 且仅写权限。仅恢复时需要读取权限。
- 审计日志可能被传唤。确保其中不包含任何密钥材料。
- `EncryptedKey` 中的 `eoaAddress` 字段是解密后的完整性检查。修改密文的攻击者无法产出匹配存储地址的有效明文。
